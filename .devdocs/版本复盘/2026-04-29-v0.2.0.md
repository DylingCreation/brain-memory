## 二、亮点

### 2.1 版本规划方法首次跑通全流程

v0.2.0 是第一个完整执行「摸底 → 可行性调查 → 版本规划 → 分批执行 → 质量检查 → 发布 → 留痕 → 复盘」闭环的版本。方法论（METHODOLOGY.md）从 MedReg KB 项目提炼出来后，第一次在本项目得到完整验证。结果证明这套方法有效——7 个功能零返工、质量红线全通过。

### 2.2 优雅降级一次到位

F-3 是本版本风险最高的功能（需重构 `processTurn` 执行流），但方案设计中先把降级逻辑写清楚，再逐步替换，最终一次性通过 12 个测试，无返工。降级效果矩阵的引入让「LLM 不可用时什么能用、什么不能用」一目了然。

### 2.3 测试零回归

从 v0.1.9 的 263 passed 到 v0.2.0 的 296 passed，新增 33 个测试用例，全量回归 0 失败。特别是 F-4 结构化日志替换了 10 个文件 50+ 处 `console.*`，在如此大规模的代码替换下仍保持零回归，证明了「渐进式替换 + 每批验证」策略的有效性。

### 2.4 SQL 安全审计建立安全基线

F-7 逐项审计 89 处 `.prepare()` + 8 处 `.exec()`，3 种动态 SQL 构造模式（scope 子句、IN 占位符、FTS5 MATCH）逐一验证。这是本项目首次完成全量 SQL 安全审计，为后续版本建立了安全基线。

### 2.5 留痕规范首次完整执行

16 份内部留痕文件，按「版本规划/ → 项目摸底/ → 开发记录/ → 问题修复/ → 技术决策/ → 性能基准/」分类存放，INDEX.md 总索引，全部真实反映实际情况。INDEX.md 在过程中发现了 2 处索引错误并当场修复，证明索引机制本身有自纠能力。

## 四、经验沉淀

### 4.1 已验证有效的模式（可复用）

| 模式 | 适用场景 | 复用时机 |
|------|---------|---------|
| **摸底 → 可行性 → 规划 → 执行 → 验证 → 发布 → 留痕** | 任何新版本启动 | 每次版本迭代 |
| **渐进式替换**（先加后删，每批验证） | 大规模代码改造（如 50+ 处修改） | 日志替换、接口重构等 |
| **降级效果矩阵** | 依赖外部服务的功能 | 任何涉及 LLM/API 降级的设计 |
| **已知问题承接表** | 版本规划时明确问题归属 | 每次版本规划 |
| **质量红线检查清单** | 发布前最后验证 | 每次发布前 |

### 4.2 新识别的风险（更新 METHODOLOGY §6.3）

| 编号 | 风险 | 等级 | 缓解措施 |
|------|------|------|---------|
| TR-07 | 整体测试覆盖率低（57%），老代码盲区大 | 高 | v0.3.0 专项补测 |
| TR-08 | JSDoc 覆盖率低（30%），文档生成质量差 | 中 | v0.3.0 专项补 JSDoc |
| TR-09 | 测试目录结构混乱（`test/` + `tests/unit/` 并存） | 低 | v0.3.0 统一结构 |

### 4.3 方法论层面的教训

1. **覆盖率测量应前置**——不应等到发布前才第一次测覆盖率。应在摸底阶段（版本启动前）就完成覆盖率测量，纳入摸底报告。
2. **留痕索引需要自动化**——手动维护 INDEX.md 容易出错（本次出现重复条目和漏索引），应考虑写一个简单的脚本来自动扫描 `.devdocs/` 目录并生成 INDEX.md。
3. **性能基线应更早建立**——v0.2.0 的性能数据是首次建立的基线，之前的版本完全没有性能数据。建议每次摸底时都跑一次性能基准。
