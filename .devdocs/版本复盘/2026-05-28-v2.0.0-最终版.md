## 二、亮点

### 2.1 审计驱动的修复流程

先做完整认知金字塔分析（5 层 281 行报告），再与项目负责人校准设计构想（澄清 6 个偏差、新增 3 个），最后按审核清单逐条修复。与传统"边看边改"模式的关键区别：

| 维度 | 传统模式 | 本次审计模式 |
|------|---------|-------------|
| 发现问题的方式 | 运行时错误 / code review 发现 | 从设计构想出发，按 5 层结构化扫描 |
| 设计 vs 实现的差距 | 靠经验判断 | 逐条对照设计构想与代码，标注偏差/风险/待确认 |
| 修复优先级 | 凭感觉 | P0/P1/P2/P3 四档分级，审核确认后执行 |
| 遗漏风险 | 高（只修看到的问题） | 低（全部 15 项逐条追踪） |
| 文档产出 | 通常没有或只有 commit message | 281 行分析报告 + 4 份留痕文档 |

### 2.4 架构澄清

LanceDB 从混乱的双 `IStorageAdapter` 实现状态回归正确角色：

| 修复前 | 修复后 |
|--------|--------|
| `LanceDBStorageAdapter implements IStorageAdapter`（40+ 方法，communities/messages/reflections = false） | 标记 `@deprecated` |
| `ContextEngine` 有 `if (config.storage === 'lancedb')` 分支 | 移除分支，统一走 SQLite |
| LanceDB 使用路径不明确 | LanceDB 仅通过 `ISearchIndex` 注入（`setSearchIndex()` → `_getSemanticSeeds()`） |

### 2.5 错误码体系

从 0 到 1：`BrainMemoryError` 基类 + 6 子类（`ConfigError` / `StorageError` / `LLMError` / `EmbeddingError` / `ValidationError` / `RuntimeError`），`ContextEngine` 构造函数已接入，导出到 `index.ts`。调用方可按 `instanceof StorageError` 做差异化处理。

## 四、经验沉淀

1. **认知金字塔审计模式可复用**：5 层（代码→模块→数据流→架构→项目）的结构化扫描比随机 code review 更系统，能发现跨层次的设计偏差（如 D7 从代码级的 `@deprecated` 标记延伸到架构级的存储层设计澄清）。

2. **审核清单逐条确认模式防止遗漏**：15 项偏差/风险/改进编号化管理，每项标注优先级（P1/P2/P3）、涉及文件、建议处置、审核结果。修复阶段对应编号执行，不留遗漏。

3. **"分析→校准→修复"优于"边看边改"**：本次先出完整分析报告（含 6 个偏差标注 + 5 个待确认），与项目负责人校准后再动手。校准过程发现 3 个误判（D1/D6/R1），避免了无效修改。

4. **设计构想是最重要的参考基准**：本次所有偏差判断都以项目负责人的设计构想为准绳（而非代码自身的自洽性）。召回流程的校准（D8/D9/D10）完全来自设计构想与代码的逐点对照。

5. **测试隔离性是 CI 稳定性的前提**：`:memory:` SQLite 消除了文件 DB 锁争用，是本次测试从 6 失败 → 0 失败的关键。

6. **POC 代码需要明确的废弃流程**：LanceDBStorageAdapter 从 POC 变为实际存在的类、出现在 `ContextEngine` 构造函数中、被用户可能误选——整个过程中缺乏"POC 完成→正式方案确定→POC 代码清理"的流程。
