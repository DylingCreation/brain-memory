## 五、经验提炼

### 做得好的

1. **外部反馈驱动的版本规划** — 不是从假设出发，而是从真实用户的生产部署日志出发。log/ 中的 5 个诊断脚本 + 3 份报告 = 最高质量的 bug 来源。

2. **PR 级别的自底向上修复** — 先 llm.ts (底层端点) → 再 tsconfig (编译层) → 再 openclaw-wrapper (集成层) → 再 plugin.core (应用层)。每层独立验证，不跨层依赖。

3. **渐进式类型安全** — `strict: false` → `strictNullChecks: true` → 未来 `strict: true`。一步到位风险大，渐进式让每步都可控。

4. **文档与代码同步修正** — METHODOLOGY.md 不是在发布后被动更新，而是在修复代码的同时修正文档（F-1 代码 + F-2 文档同日完成）。

5. **留痕密度** — 10 项任务全部有独立的开发记录，复盘时 100% 可追溯。

### 踩过的坑

1. **CHANGELOG 误判为"空"** — 第一次摸底只读了文件头部，看到 `[Unreleased]` 就下结论。实际后续有 30+ 版本的完整记录。**教训: 大文件必须先 `tail` 确认尾部内容。**

2. **npm test 2 次失败误判为回归** — LanceDB POC 的 forks 竞态 flake 在多次运行中不稳定出现。**教训: 不要只看失败计数，要看失败的测试文件名是否与变更模块相关。**

3. **openclaw-plugin.json 有两个不同的修复版本** — new/ 中的修复与 v1.8.0 F-8 的修复是独立进行的，需要人工对照才能确认无遗漏。**教训: 外部反馈应该有一个统一的 backlog 追踪文件 (如 UPSTREAM-BACKLOG.md)，避免并行修复导致遗漏。**

### 可改进的

1. **留痕应该边做边写** — F-9/F-10 的留痕是最后统一补的，不如开发时同步写。

2. **lint 应该在每次提交前跑** — 最后的 lint error 是在发布检查清单中才发现，如果每次 commit 前跑可以更早发现。

3. **性能基准 flake 需要根治** — `perf-tiered-benchmark.test.ts` 从 v1.6.2 起就是已知 flake，每次发布都看到它。不应该无限期 defer。
