## 一、版本目标回顾

v0.2.0 解决了「能用不崩溃」（可靠生产），v1.0.0 解决两个根本问题：

1. **核心架构补全**——图 Schema 残缺、LLM 依赖过重、插件集成不完整、记忆共享缺失、retriever 半成品未接入
2. **工程质量夯实**——覆盖率 57%→目标 70%+，JSDoc 30%→目标 80%，.gitignore/.npmignore 清理，测试目录统一

## 四、亮点

1. **覆盖率 57% → 83.2%**：补测 7 个测试文件 229 用例，P0 核心模块（plugin/core、llm、embed、recaller）全部 > 90%
2. **JSDoc 从几乎空白到 50.7%**：store.ts + analyzer.ts 全量注释，核心 API 皆有文档
3. **.gitignore 490→61 行**：删除 400+ 行 AI 批量生成的无效规则（neo4j/stripe/discord 等不相关项目）
4. **Git 历史重构**：53 个混乱 commit → 16 个 clean commit，从头到尾格式统一、可追溯
5. **召回性能优秀**：avg 0.44ms / p95 0.50ms（200 节点），远低于 500ms 目标
6. **Qwen thinking 优化**：延迟从 ~4s 降到 ~1s（实测 4 倍提升）

## 五、问题与根因

| 问题 | 根因 | 改进措施 |
|------|------|---------|
| **测试用例 446→363 丢失** | B-5 批次 5 个测试文件（115 用例）从未 git commit，留痕写了但代码未落盘 | 留痕必须与代码提交同步；开发记录中"测试全通过"必须对应已推送的 git commit |
| **远程更新与本地脱节** | 远程 2 个新提交（Ollama/Qwen/Windows），本地未同步就开始工作 | 版本启动前 `git fetch` + `git diff` 确认远程状态 |
| **JSDoc 原目标 80% 过高** | 203 个导出项中大量内部类型定义，80% 不现实 | 目标务实调整：先 35%（核心 API），后续版本再冲 80% |
| **TR-10/TR-11 未处理** | 启发式提取质量验证 + Reranker 降级路径测试，工作量超出版本预期 | 纳入 v1.1.0 规划，标注风险等级 |
| **B-5 用例数下降未及时发现** | 留痕文档中 446 用例被当作基线引用，未与仓库实测交叉验证 | 基线数据必须以 `vitest run` 实测为准，不能仅依赖留痕 |
