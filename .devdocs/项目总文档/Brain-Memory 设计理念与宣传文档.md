# Brain-Memory：AI 的"大脑级"长期记忆引擎

> **核心理念**：让 AI 像人一样记忆——不仅记住，还会遗忘、反思、推理、融合知识。
>
> 面向读者：AI 开发工作者 + 普通 AI 用户
>
> 基于：[认知金字塔分析报告](./brain-memory-认知金字塔分析报告.md) · [骨架地图 7 份文档](./.devdocs/项目总文档/)

---

## 一句话解释

Brain-Memory 是一个 **TypeScript 记忆引擎**。它让 AI Agent 拥有"长期记忆"——记住你的偏好、学到的技能、踩过的坑、做出的决策，而不是每次对话都从零开始。

---

## 一、它模拟了人类记忆的四种能力

```
       👤 人类记忆                    🤖 Brain-Memory

  ┌─────────────────┐         ┌──────────────────┐
  │ 记住重要的事      │   ←→    │ 8 类记忆自动分类存储 │
  │ 遗忘不重要的事    │   ←→    │ Weibull 衰减模型    │
  │ 事后反思总结      │   ←→    │ 双级反思引擎        │
  │ 举一反三推理      │   ←→    │ 知识图谱推理引擎     │
  └─────────────────┘         └──────────────────┘
```

### 1.1 记住——8 类记忆自动分类

不需要手动整理。AI 从对话中自动提取并分类：

| 类别 | 记住什么 | 例 |
|------|---------|-----|
| 👤 **用户画像** | 你的身份、角色、背景 | "我是一个 Python 后端开发者" |
| 🎯 **偏好习惯** | 你喜欢/讨厌什么 | "我更喜欢用 pnpm 而非 npm" |
| 📦 **实体信息** | 项目、工具、环境的客观事实 | "项目部署在阿里云 ECS 上" |
| ⚡ **事件/报错** | 发生过的问题和异常 | "上次 Docker 构建因内存不足失败" |
| ✅ **完成的任务** | 做过什么、讨论过什么 | "上周完成了 API 限流模块" |
| 🔧 **可复用技能** | 怎么做事的具体方法 | "使用 docker-compose up -d 启动服务" |
| 📋 **案例经验** | 具体场景的成功/失败案例 | "上次用 Redis 做缓存层解决了查询瓶颈" |
| 🔮 **模式规律** | 跨案例的抽象规律 | "每次部署前跑一遍集成测试能避免 80% 的问题" |

### 1.2 遗忘——该记住的不会丢，该忘的自然消失

不是简单的时间过期，而是 **Weibull 生存函数**模拟的三层衰减：

```
核心知识 (importance > 0.7)
  └─ 衰减极慢 → 几乎永久保留
  例："用户的编程语言偏好"

工作知识 (importance > 0.4)
  └─ 中等速度衰减 → 长期不用才淡化
  例："三个月前临时用的一个脚本"

边缘知识 (importance ≤ 0.4)
  └─ 快速衰减 → 自动清理噪音
  例："某次对话中提到的一句无关信息"
```

**动态记忆 3 倍加速**：有时效性的信息（如"当前版本号""今天的状态"）比稳定知识（如"设计模式定义"）遗忘速度快 3 倍。

### 1.3 反思——对话结束后自动总结洞察

每轮对话不只是"存储"，而是像人一样坐下来想想"这次对话让我学到了什么"：

```
会话结束
  │
  ├─ 用户画像洞察: "用户对 TypeScript 类型安全要求很高"
  ├─ Agent 教训:     "今天推荐的 Docker 方案在 NVIDIA 环境下不兼容"
  ├─ 经验提炼:       "GPU 相关部署需要额外检查 nvidia-docker 驱动"
  └─ 重要决策:       "确认 SQLite 为唯一真值存储，LanceDB 为必需伴随语义索引"
```

这些反思结果**也被存为知识节点**，参与后续的搜索和推理——不是静态条目录，而是动态生长的知识图谱。

### 1.4 推理——从已知推到未知

当记忆足够多时，系统能够发现隐藏的关联：

| 推理类型 | 行为 | 示例 |
|---------|------|------|
| **路径推导** | 发现间接关联 A→B→C | "用户用 Docker → Docker 依赖 GPU → 所以需要 nvidia-docker" |
| **隐含关系** | 共享邻居暗示隐藏连接 | "项目 A 和项目 B 都使用了 Redis → 可能存在数据层共享需求" |
| **模式泛化** | 多次失败总结规律 | "连续 3 次部署超时 → 部署流程存在系统性问题，不是偶发" |
| **矛盾检测** | 发现冲突信息 | "节点 A 说数据库是 MySQL，节点 B 说数据库是 PostgreSQL" |

## 二、它如何工作——三条管道

### 管道一：学习 (processTurn) —— 从对话中提取知识

```
用户消息进入
  → 噪声过滤（跳过"好的""嗯""哈哈"）
  → 三级提取:
     Tier 1: 规则引擎（匹配代码块/命令/8类正则） ← 零 LLM 成本，<10ms
     Tier 2: LLM 深度理解（提取结构化知识节点）   ← 有 LLM 配置时
     Tier 3: 容错解析（LLM 输出格式异常时自动修复） ← 最后防线
  → 合并去重（LLM 结果优先，启发式补充）
  → 生成向量嵌入 → 存入 SQLite 数据库
  → 更新知识图谱的边关系
  → 更新"工作记忆"（当前关注什么、在做什么任务）
```

**关键设计**：LLM 不可用时不崩溃——自动降级到 Tier 1，仍然能提取粗粒度知识。

### 管道二：回忆 (recall) —— 查询相关记忆

```
用户提问
  → 信息量判断（"好的"跳过，"怎么部署？"触发）
  → 意图分析（技术问题？偏好查询？时间敏感？）
  → 查询扩展（"部署"自动补成"部署 deploy release 发布"）
  → 四路并行搜索:
     ① 关键词 + 向量搜索 → 社区扩展
     ② 社区语义匹配 → 成员召回
     ③ LanceDB 语义搜索（必需伴随索引）
     ④ 外部长期记忆（USER.md/MEMORY.md）
  → 统一去重 + PageRank 排序
  → 多路径增强（同时被多条路径命中的记忆 ×1.2 权重）
  → 时间衰减 + 时间敏感加权（"最近""最新"触发生鲜度增强）
  → 返回 Top-N 最相关记忆
```

### 管道三：维护 (maintenance) —— 离线整理知识图谱

```
定期触发（会话结束时自动运行）
  → 去重（发现语义重复的知识，合并）
  → PageRank 排序（重新计算节点重要性）
  → 社区检测（根据连接密度自动分组——"Docker 相关技能""Python 相关错误"）
  → 社区摘要（用 LLM 给每组写一句主题描述）
  → 衰减归档（低质量孤立记忆自动标记 deprecated）
```

**增量维护**：如果只有少量节点变更（< 10%），只重组变更节点所在的子图，性能提升 5 倍以上。

## 三、核心设计理念

### 3.1 知识图谱 + 向量搜索双引擎

```
        ┌─────────────────────────────────────┐
        │           混合召回引擎               │
        │                                      │
        │   ┌──────────┐    ┌──────────────┐  │
        │   │ 知识图谱   │    │  向量搜索     │  │
        │   │ PageRank  │    │  余弦相似度   │  │
        │   │ LPA 社区  │    │  FTS5 全文   │  │
        │   │ 图遍历    │    │  LanceDB     │  │
        │   └─────┬────┘    └──────┬───────┘  │
        │         │                │           │
        │         └───────┬────────┘           │
        │                 ▼                    │
        │         RRF 分数融合 + 统一排序       │
        └─────────────────────────────────────┘
```

**不是简单的 RAG**。向量搜索告诉你"这段话和查询有多相似"，知识图谱告诉你"这段知识和哪些其他知识有关联"。两者互补——向量负责精准命中，图谱负责关联发散。

### 3.2 六层 Scope 隔离

不同平台、不同 Agent、不同会话的记忆自然隔离：

```
platform: discord → workspace: /project-a → agent: code-assistant → user: yang → chat: channel-1 → thread: topic-3
    ↓                    ↓                      ↓                      ↓              ↓               ↓
  哪个平台              哪个项目                 哪个 AI                  哪个用户        哪个群/会话      哪个子话题
```

六层由粗到细，查询时前缀匹配——查"discord + agent: code-assistant"能覆盖该 agent 在该平台上的所有细粒度记忆。

### 3.3 零 LLM 仍可工作，LanceDB 可随时重建

设计原则：**可选依赖（LLM/Embedding/Reranker）不可用时，系统降级但不崩溃。必需依赖（SQLite/LanceDB）通过可重建性保证数据安全。**

```
全功能 (LLM ✓  Embed ✓  LanceDB ✓  Reranker ✓)
  │
  ├─ 无 Reranker       → 余弦相似度回退
  ├─ 无 Embedding      → FTS5 纯文本搜索
  └─ 无 LLM            → 仅启发式提取 + FTS5 召回
                          仍然可用，只是不如全功能精确

LanceDB 是必需的伴随语义索引——但它不存真值，
仅存 embedding → nodeId 映射。任何时候出问题，
从 SQLite 全量 rebuild() 即可恢复。
```

最少跑起来的配置只需要 **SQLite + LanceDB**——两个文件，零外部服务。

### 3.4 知识融合：自动去重和关联

当知识图谱增长到一定规模（≥20 节点 + ≥3 社区），系统自动检测：

- **重复知识**：同一个概念被多次以不同表述记录 → 自动合并
- **相关但独立**：两个社区的节点语义相近但不应合并 → 自动建立关联边

决策由 LLM 辅助（不可用时用启发式阈值回退），执行结果永久写入图谱。

### 3.5 工作记忆：知道"现在在做什么"

不依赖 LLM，纯从结构化提取结果推导：

```
当前任务: [修复 Docker 部署问题, 编写 API 文档]
最近决策: [用 Redis 替代本地缓存, 将日志级别改为 WARN]
约束偏好: [用户偏好: Docker 部署而非裸机, 代码风格: Prettier default]
AI 承诺:   [我会帮你检查 nvidia-docker 驱动版本]
当前关注: 用户正在排查 Docker GPU 相关问题...
```

这个上下文在每次对话中自动维护，注入到系统提示词中——让 AI 不会忘记"我们刚才在聊什么"。

## 四、技术实现

### 4.1 技术栈

| 组件 | 选型 | 选型理由 |
|------|------|---------|
| 语言 | TypeScript 5.9 (strict) | 强类型适合复杂数据模型 |
| 主数据库 | SQLite (同步 API) | 零配置、嵌入式、无需独立服务 |
| 语义索引 | LanceDB（必需） | 向量搜索加速，非真值存储，可随时从 SQLite 全量重建 |
| LLM 端点 | OpenAI / DashScope / Ollama / Anthropic | 多端自动路由，所有端点统一接口 |
| Embedding | OpenAI-compatible / Ollama | 统一抽象接口 |
| 测试 | Vitest 3.2 (847 用例) | 测试代码行数 (13,019) > 源码行数 (10,642) |
| 宿主 | OpenClaw 插件体系 | 5 个生命周期钩子集成 |

### 4.2 关键算法

| 算法 | 用途 | 实现 |
|------|------|------|
| **Weibull 衰减** | 记忆遗忘 | `exp(-(t/λ)^β)`，λ 从半衰期推导，三层 β 值区分核心/工作/边缘 |
| **个性化 PageRank** | 召回排序 | 标准迭代 + 60s 图缓存 + 增量重算(脏节点 < 10%) |
| **LPA 社区检测** | 知识分组 | 标签传播算法，≤50 轮迭代，冻结干净节点实现增量更新 |
| **LSH 去重** | 语义重复检测 | 8-bit 签名分桶，O(n²) → O(n × bucket) |
| **RRF 融合** | 多路召回合并 | Reciprocal Rank Fusion (k=60) |
| **双向 Jaccard** | 名称相似度 | 中英文分词 + 集合交并比 |

### 4.3 性能数据 (10K 节点规模)

| 操作 | 耗时 | 吞吐 |
|------|------|------|
| PageRank (全量) | ~1.1s | — |
| PageRank (增量 5% dirty) | ~0.1s | — |
| LPA 社区检测 (全量) | ~0.7s | — |
| 节点插入 | — | ~24,759 ops/s |
| 核心召回 | ~0.44ms avg | — |
| 增量加速比 | ≥5x (PR) / ≥3x (LPA) | — |

## 五、为 AI 开发者：一分钟集成

### 作为 OpenClaw 插件

```bash
npm install memory-likehuman-pro
```

在 `openclaw.json` 中：

```json
{
  "plugins": {
    "entries": {
      "brain-memory": {
        "config": {
          "dbPath": "~/.openclaw/brain-memory.db",
          "mode": "full",
          "llm": { "baseURL": "http://localhost:11434", "model": "qwen3:14b" },
          "embedding": { "baseURL": "http://localhost:11434", "model": "nomic-embed-text" }
        }
      }
    }
  }
}
```

无需任何代码——插件自动注册到消息管道，提取、召回、维护全自动。

### 作为独立库

```typescript
import { ContextEngine } from 'memory-likehuman-pro';

const engine = new ContextEngine({
  dbPath: ':memory:',
  storage: 'sqlite',
  mode: 'lite',  // 无需 LLM 也能用
});

// 写入知识
await engine.processTurn({
  sessionId: 'demo',
  agentId: 'assistant',
  workspaceId: 'my-project',
  messages: [
    { role: 'user', content: '我需要部署一个 Python Flask 应用到 Docker' }
  ],
});

// 查询记忆
const result = await engine.recall('Docker 部署');
console.log(result.nodes.map(n => n.name));
// → ['任务-部署Python Flask到Docker', '技能-docker compose up', '实体-项目部署环境']
```

### 最小可用配置

```json
{ "dbPath": "brain-memory.db", "storage": "sqlite", "mode": "lite" }
```

SQLite + LanceDB 为必需依赖，LLM / Embedding 可选。无 LLM 时提取用规则引擎，召回用文本搜索。

## 六、为 AI 用户：它记住了什么，你感受到什么

### 你会注意到的变化

| 之前 (无记忆) | 之后 (Brain-Memory) |
|-------------|-------------------|
| "帮我写个 Dockerfile" | "根据你项目已有的 docker-compose 配置，我帮你生成匹配的 Dockerfile" |
| "上次那个 bug 怎么修的？" | "上次是 nvidia 驱动版本不匹配，我直接帮你查当前驱动是不是 535 版本" |
| "我不喜欢 YAML" | 记一次，下次全部输出 JSON |
| 每次问都要重复上下文 | 跨会话自动关联——知道你在做的项目和已经确定的决策 |
| 三天前的对话完全忘记 | 核心知识永久保留，边缘信息自然衰减 |

### 你不会注意到的（内部机制）

- 🧹 聊天中的寒暄、表情包、单字回复不会变成"知识"（噪声过滤）
- 🗑️ 三个月前偶然提到的无关信息自动消失（三层衰减）
- 🔗 同一件事被多次提到不会产生重复条目（自动去重和融合）
- 🏝️ 和 AI-A 的对话不会泄漏到 AI-B（六层 Scope 隔离）
- 🔄 即使 LLM 服务挂了，记忆系统还在工作（降级设计）

## 七、设计理念总结

| 原则 | 含义 | 实现 |
|------|------|------|
| **像人一样记忆** | 不仅是存储，还要遗忘、反思、推理 | Weibull 衰减 + 反思引擎 + 推理引擎 |
| **渐进增强** | 最少只需 SQLite，加 LLM/Embedding 逐步获得更强能力 | 三级提取 + 五级降级 |
| **知识结构化** | 不是一堆文本片段，是多维连接的知识图谱 | 8 类节点 + 11 种边 + 社区检测 |
| **关注点隔离** | 不同场景的记忆自然分开，不互相污染 | 六层 Scope + 跨域共享策略 |
| **自我维护** | 不需要手动管理记忆库 | 自动去重 + 融合 + 衰减归档 |
| **可观测** | 所有行为可追踪、可验证 | 847 测试覆盖 + 详细日志 + Web 控制面板 |
| **工程可靠** | 方法论驱动而非经验驱动 | 摸底→规划→开发→质检→发布→留痕→复盘 全闭环 |
