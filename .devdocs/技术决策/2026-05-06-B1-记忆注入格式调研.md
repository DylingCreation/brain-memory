## 一、现状分析

### 1.1 当前注入链路

```
OpenClaw before_message_write 钩子
  → 从 sessionMemoryCache 读取 cachedMemoryContext
  → 附加到 rawEvent.memoryContext
  → 返回（同步，不阻塞消息发送）
```

**问题**：缓存的 `memoryContext` 是 recall 返回的原始数据结构，**未经过格式化**直接附加。Agent 收到的是一个包含 `relatedNodes` 数组的 JSON 对象，没有结构化模板、没有 token 预算控制。

### 1.2 assembleContext 现有能力

`src/format/assembleContext.ts` 已经具备相当完善的能力：

| 能力 | 当前实现 | 状态 |
|------|---------|------|
| **Token 预算** | `tokenBudget` 参数，0=无限，>0 按预算截断 | ✅ 已有 |
| **三种策略** | full（完整 XML）/ summary（仅名称+描述）/ adaptive（>6 节点自动 summary）/ off（跳过） | ✅ 已有 |
| **语言感知估算** | `estimateNodeTokens()` 中文 1.8ch/token、英文 2.5ch/token | ✅ 已有 |
| **排序** | recalled 优先 > active > 按类型(SKILL>TASK>EVENT) > validatedCount > pagerank | ✅ 已有 |
| **社区分组** | 按 communityId 分组，附 LLM 生成的社区摘要 | ✅ 已有 |
| **情景记忆** | top 3 节点拉取原始对话片段，智能截断 400ch | ✅ 已有 |
| **系统提示生成** | `buildSystemPromptAddition()` 生成总结性提示 | ✅ 已有 |

### 1.3 核心问题：assembleContext 未被 OpenClaw 链路调用

**关键断裂**：`assembleContext` 设计精但**仅在独立库模式下使用**。OpenClaw 的 `before_message_write` 钩子直接把缓存的原始 `memoryContext`（`{relatedNodes: [...]}`）附加到事件上，没有经过 `assembleContext` 格式化。

### 1.4 tokenBudget 的来源

当前 `assembleContext` 接受 `tokenBudget: number` 参数，但**没有任何地方计算合适的预算值**。规划要求的「不超过上下文 20%」需要在调用方实现。

---

## 二、方案设计

### 2.1 核心思路

**不重写 assembleContext，而是将它接入 OpenClaw 链路。**

```
before_message_write 钩子
  → 读取 cachedMemoryContext（已有）
  → 调用 assembleContext(db, {tokenBudget, recallStrategy, nodes, edges})
  → 将格式化后的 xml/systemPrompt 附加到事件
```

### 2.2 Token 预算计算策略

根据 Agent 使用的主流模型，预设上下文窗口大小：

| 模型 | 上下文窗口 | 20% 预算 |
|------|-----------|---------|
| qwen3.6-plus | 32,768 tokens | ~6,500 tokens |
| qwen3-8b | 32,768 tokens | ~6,500 tokens |
| gpt-4o-mini | 128,000 tokens | ~25,600 tokens |

**默认预算**：`tokenBudget = 0`（不限制），但提供配置项。用户可手动设置 `memoryBudgetTokens`。

**建议的默认值**：6,000 tokens（适配 32K 上下文窗口，留有余量）。

### 2.3 配置项设计

在 `BmConfig` 中新增：

```typescript
memoryInjection: {
  enabled: boolean;         // default true
  strategy: "full" | "summary" | "adaptive" | "off";  // default "adaptive"
  tokenBudget: number;      // default 6000 (0 = unlimited)
  maxNodes: number;         // default 12（上限，即使预算允许也不超过）
  includeEpisodic: boolean; // default true（是否附原始对话片段）
}
```

### 2.4 实施步骤

| 步骤 | 内容 | 产出 |
|------|------|------|
| 1 | 在 `BmConfig` 和 `DEFAULT_CONFIG` 中新增 `memoryInjection` 配置 | 类型定义 + 默认值 |
| 2 | 修改 `before_message_write` 钩子，调用 `assembleContext` 格式化 | 代码变更 |
| 3 | 格式化结果通过 `memoryContext` 传递（结构化 XML + systemPrompt） | 代码变更 |
| 4 | 编写测试用例（token 预算、三种策略、边界条件） | 测试用例 |
| 5 | 实测验证：注入后的 token 占比 ≤ 20% | 验证报告 |

## 三、风险评估

| 风险 | 等级 | 预案 |
|------|------|------|
| assembleContext 依赖 DatabaseSyncInstance，OpenClaw 钩子中需获取 DB 实例 | 中 | 通过 pluginInstance.db 获取，已有路径 |
| 格式化后的 XML 体积可能超出预期 | 低 | tokenBudget + maxNodes 双重保护 |
| 同步钩子中执行 assembleContext 可能增加延迟 | 低 | assembleContext 为纯本地操作（无 LLM），< 5ms |

---

## 四、结论

**改造难度：🟢 低**

assembleContext 已有 90% 的能力，只需接入 OpenClaw 链路 + 新增配置项 + 完善 token 预算计算。预估工时：调研 0.25 天 + 开发 1 天。
