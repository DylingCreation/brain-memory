# 否决 vitest 3.2.4 → 4.1.6 升级

| 字段 | 内容 |
|------|------|
| **日期** | 2026-05-20 |
| **状态** | Accepted |
| **决策人** | 代码助手 + 老板 |
| **关联版本** | v1.6.0 |

## 背景

v1.6.0 C-1 批次目标为消除 deprecated 依赖。vitest 当前版本 3.2.4，最新版 4.1.6。升级前需要评估可行性与收益。

## 调研过程（实测数据）

### vitest 3.2.4（当前基线）

| 维度 | 数据 |
|------|------|
| 不稳定率（5轮全量测试） | ~20%（5轮中1轮有1个失败） |
| 失败原因 | 随机 SQLite `database is locked` |
| 总测试数 | 623 |
| 全绿概率 | 80% |

### vitest 4.1.6（实测）

| 维度 | 数据 |
|------|------|
| 不稳定率（5轮全量测试） | **~80%+**（5轮中4轮有2~4个失败） |
| 失败原因 | 随机 SQLite `database is locked` |
| 总测试数 | **616**（少了 7 个测试用例） |
| 全绿概率 | <20% |

### 根因分析

vitest 4 migration guide 明确写了：
- `poolOptions` is **removed**，所有 pool 选项改为顶层级
- `maxForks` / `minForks` 重命名为 `maxWorkers` / `minWorkers`
- Custom pool interface 被 **rewritten**

brain-memory 的 vitest 配置依赖 `poolOptions.forks.maxForks=4, minForks=2` 来控制 SQLite 文件并发。vitest 4 下该配置全部失效，使用自己的默认并发策略，导致 fork 进程间 SQLite 锁竞争恶化 4 倍。

### vitest 4 收益评估

vitest 4 的核心新特性全部面向浏览器测试（Browser Mode 稳定化、Visual Regression、Playwright Traces、Locator 增强），brain-memory 是纯 Node.js 库，**这些特性对项目零价值**。TypeScript 类型推断增强（`test.extend`）是项目可能用到的唯一特性，但代码中未深度使用该模式。

### 收益/风险比

- **收益**：≈ 零（核心价值全在浏览器测试，项目用不到）
- **风险**：灾难级（测试不稳定，SQLite lock 恶化 4x，丢失 7 个测试用例）
- **结论**：严重负数，否决升级

## 决策

**vitest 保持 3.2.4，不升级到 4.x。** 待 vitest 后续版本修复 pool 层对 SQLite 的兼容性后重新评估。

## 影响

- `package.json` 中 vitest 与 @vitest/coverage-v8 锁定在 3.2.4
- C-1 批次 "6 deprecated 依赖" → "5 升级 + 1 否决"
- 后续如 vitest 5 修复了 pool 层兼容性，可在此 ADR 基础上重新评估
- vitest 3.2.4 安全更新仍需跟进
