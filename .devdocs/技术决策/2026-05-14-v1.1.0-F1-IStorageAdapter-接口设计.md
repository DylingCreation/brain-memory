## 背景

当前所有存储操作紧耦合到 `DatabaseSyncInstance`（SQLite）。需要抽象出 `IStorageAdapter` 接口，将存储与算法解耦，为后续 LanceDB 等后端铺路。

## 设计原则

1. **接口最小化**：只暴露调用方需要的操作，不暴露存储实现细节
2. **存储无关**：接口不涉及 SQL、SQLite 概念
3. **向后兼容**：`SQLiteStorageAdapter` 实现后，调用方代码改动最小
4. **可扩展**：为增量图维护预留"脏标记"相关接口

## 接口设计

### 核心思路

将当前 `store/store.ts` 的函数 + `graph/` 模块对存储的访问统一抽象为一个接口。调用方（ContextEngine、Recaller、Maintenance 等）通过接口操作，不直接持有 `DatabaseSyncInstance`。

### IStorageAdapter 接口

```typescript
/**
 * IStorageAdapter — brain-memory 存储层抽象接口
 * 
 * 所有 CRUD、图查询、向量搜索、消息操作通过此接口进行。
 * 具体实现：SQLiteStorageAdapter（v1.1.0 首批），LanceDBStorageAdapter（v1.3.0 POC）。
 */
export interface IStorageAdapter {
  // ─── 生命周期 ───────────────────────────────────────────
  
  /** 初始化存储（创建表、索引、运行迁移）。对应当前 initDb() */
  initialize(): void;
  
  /** 关闭存储，释放资源。对应当前 engine.close() */
  close(): void;
  
  /** 检查存储连接是否正常 */
  isConnected(): boolean;
  
  // ─── 节点 CRUD ──────────────────────────────────────────
  
  /** 按名称查找节点。返回 null 如果不存在 */
  findNodeByName(name: string): BmNode | null;
  
  /** 按 ID 查找节点。返回 null 如果不存在 */
  findNodeById(id: string): BmNode | null;
  
  /** 插入或更新节点（按名称匹配）。返回节点和是否新建 */
  upsertNode(input: NodeUpsertInput, sessionId: string): { node: BmNode; isNew: boolean };
  
  /** 标记节点为 deprecated（软删除） */
  deprecateNode(nodeId: string): void;
  
  /** 合并两个节点：保留 validatedCount 高的，合并 sourceSessions 和边 */
  mergeNodes(keepId: string, mergeId: string): void;
  
  /** 查询所有活跃节点，可选范围过滤 */
  findAllActive(filter?: StorageFilter): BmNode[];
  
  /** 获取所有边 */
  findAllEdges(): BmEdge[];
  
  /** 批量更新节点 pagerank 分数（事务） */
  updatePageranks(scores: Map<string, number>): void;
  
  /** 批量更新节点 community_id（事务） */
  updateCommunities(labels: Map<string, string>): void;
  
  /** 更新节点访问计数 */
  updateAccess(nodeId: string): void;
  
  // ─── 边 CRUD ────────────────────────────────────────────
  
  /** 插入或更新边（按 fromId + toId + type 匹配） */
  upsertEdge(input: EdgeUpsertInput): BmEdge;
  
  /** 获取指定节点的出边 */
  findEdgesFrom(nodeId: string): BmEdge[];
  
  /** 获取指定节点的入边 */
  findEdgesTo(nodeId: string): BmEdge[];
  
  // ─── 搜索 ───────────────────────────────────────────────
  
  /** 文本搜索节点（FTS5 → LIKE 回退），返回 top-N */
  searchNodes(query: string, limit: number, filter?: StorageFilter): BmNode[];
  
  /** 获取 top-N 活跃节点（按 pagerank 排序） */
  findTopNodes(limit: number, filter?: StorageFilter): BmNode[];
  
  /** 向量搜索：返回节点及其余弦相似度分数 */
  vectorSearch(queryVec: number[], limit: number, filter?: StorageFilter): ScoredNode[];
  
  /** 图谱遍历：从种子节点出发，最多 maxDepth 跳，返回可达节点和边 */
  graphWalk(seedIds: string[], maxDepth: number): { nodes: BmNode[]; edges: BmEdge[] };
  
  // ─── 向量操作 ───────────────────────────────────────────
  
  /** 存储或替换节点的嵌入向量 */
  saveVector(nodeId: string, content: string, vec: number[]): void;
  
  /** 获取节点的嵌入向量 */
  getVector(nodeId: string): Float32Array | null;
  
  /** 获取节点向量的内容 hash（用于缓存检测） */
  getVectorHash(nodeId: string): string | null;
  
  /** 加载所有节点-向量对 */
  loadAllVectors(): Array<{ nodeId: string; embedding: Float32Array }>;
  
  // ─── 社区 ───────────────────────────────────────────────
  
  /** 存储或更新社区摘要 */
  upsertCommunity(id: string, summary: string, nodeCount: number, embedding?: number[]): void;
  
  /** 获取单个社区摘要 */
  getCommunity(id: string): CommunitySummary | null;
  
  /** 获取所有社区摘要 */
  getAllCommunities(): Map<string, CommunitySummary>;
  
  /** 删除孤儿社区摘要 */
  pruneCommunities(): number;
  
  /** 社区向量搜索 */
  communityVectorSearch(queryVec: number[], minScore: number): ScoredCommunity[];
  
  /** 按社区 ID 获取成员节点 */
  findNodesByCommunities(communityIds: string[], perCommunity: number): BmNode[];
  
  /** 获取同一社区的其他节点 */
  findCommunityPeers(nodeId: string, limit: number): string[];
  
  /** 获取每个社区的代表节点 */
  findCommunityRepresentatives(perCommunity: number): BmNode[];
  
  // ─── 消息 ───────────────────────────────────────────────
  
  /** 保存对话消息 */
  saveMessage(sessionId: string, turn: number, role: string, content: unknown): void;
  
  /** 获取未提取的消息 */
  getUnextractedMessages(sessionId: string, limit: number): MessageRow[];
  
  /** 标记消息为已提取 */
  markMessagesExtracted(sessionId: string, upToTurn: number): void;
  
  /** 获取时间相关的消息片段 */
  getEpisodicMessages(sessionIds: string[], nearTime: number, maxChars: number): EpisodicSnippet[];
  
  // ─── 统计与元数据 ──────────────────────────────────────
  
  /** 获取统计信息（节点数、边数、类别分布等） */
  getStats(): StorageStats;
  
  /** 获取 Schema 版本号 */
  getSchemaVersion(): number;
  
  // ─── 增量图维护（v1.1.0 F-3 预留） ─────────────────────
  
  /** 获取脏节点 ID 集合（由上层标记） */
  getDirtyNodes(): Set<string>;
  
  /** 标记节点为脏（图变更后调用） */
  markDirty(nodeIds: string[]): void;
  
  /** 标记边为脏（图变更后调用） */
  markDirtyEdges(edgeIds: string[]): void;
  
  /** 清除脏标记 */
  clearDirty(): void;
  
  /** 获取受脏节点影响的局部子图（用于增量 PageRank/社区检测） */
  getAffectedSubgraph(maxDepth: number): { nodes: BmNode[]; edges: BmEdge[] };
}
```

## 迁移策略

### SQLiteStorageAdapter 实现路径

1. **新文件** `src/store/adapter.ts` — 定义 `IStorageAdapter` 接口 + 辅助类型
2. **新文件** `src/store/sqlite-adapter.ts` — `SQLiteStorageAdapter` 实现
   - 内部持有 `DatabaseSyncInstance`
   - 每个方法委托到现有 `store/store.ts` 的对应函数
   - 脏标记在内存中维护（`Set<string>`）
3. **ContextEngine 改造**：
   - 构造函数从 `initDb()` 改为 `new SQLiteStorageAdapter(dbPath)`
   - `this.db` 替换为 `this.storage: IStorageAdapter`
4. **Recaller 改造**：
   - 构造函数从 `(db, cfg)` 改为 `(storage, cfg)`
   - 所有 `store/store.ts` 导入替换为 `this.storage.xxx()`
5. **graph/ 模块改造**：
   - `maintenance.ts`、`pagerank.ts`、`community.ts`、`dedup.ts` 从接收 `DatabaseSyncInstance` 改为接收 `IStorageAdapter`
   - 图算法本身（纯内存计算）不变，只有存储访问走接口

### 调用方改动量估算

| 模块 | 改动量 | 说明 |
|------|--------|------|
| `engine/context.ts` | ~30 行 | 初始化 + 直接调用替换 |
| `recaller/recall.ts` | ~40 行 | 所有 store 调用替换 |
| `graph/maintenance.ts` | ~15 行 | 参数类型替换 |
| `graph/pagerank.ts` | ~10 行 | 图加载走接口 |
| `graph/community.ts` | ~15 行 | 图加载 + 社区操作走接口 |
| `graph/dedup.ts` | ~10 行 | 向量 + 节点操作走接口 |
| **合计** | **~120 行** | 纯替换，无逻辑变更 |

## 关键决策

### 决策 1：脏标记放在接口内还是外部？

**结论**：放在接口内。

理由：
- 脏标记是存储层的状态，应由存储层管理
- 不同后端可能有不同的脏标记策略（内存 vs 持久化）
- 查询缓存的失效机制可以复用脏标记

### 决策 2：是否保留 `DatabaseSyncInstance` 类型暴露？

**结论**：不暴露。接口层不出现任何 SQLite 类型。

理由：
- 如果暴露，调用方会依赖 SQLite 特性，破坏抽象
- `getAffectedSubgraph()` 返回统一的 `{ nodes, edges }` 结构，调用方不关心底层查询方式

### 决策 3：`graphWalk` 放在接口内还是作为纯算法？

**结论**：放在接口内。

理由：
- `graphWalk` 使用递归 CTE，是数据库层面的操作
- 不同后端的图遍历实现方式不同（CTE vs 内存遍历）
- 纯算法部分（PageRank、LPA 社区检测）仍在 `graph/` 模块

## 影响

- 接口冻结后，`SQLiteStorageAdapter` 的实现是体力活（逐方法委托）
- `graph/` 模块的纯算法代码不变，仅参数类型从 `DatabaseSyncInstance` 改为 `IStorageAdapter`
- 测试不需要重写，只需 mock `IStorageAdapter` 或集成测试用 `SQLiteStorageAdapter`
