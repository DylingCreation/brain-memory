## 一、现状全景

### 1.1 当前 Schema

| 维度 | 定义 | 数量 |
|------|------|------|
| **GraphNodeType** | TASK / SKILL / EVENT | 3 种 |
| **MemoryCategory** | profile / preferences / entities / events / tasks / skills / cases / patterns | 8 类 |
| **EdgeType** | USED_SKILL / SOLVED_BY / REQUIRES / PATCHES / CONFLICTS_WITH | 5 种 |

### 1.2 核心缺陷

**缺陷 A：EDGE_TO_CONSTRAINT 全指向 SKILL**

```typescript
// 当前（所有 5 种边的 to 端都只有 SKILL）
EDGE_TO_CONSTRAINT = {
  USED_SKILL:     ["SKILL"],
  SOLVED_BY:      ["SKILL"],
  REQUIRES:       ["SKILL"],
  PATCHES:        ["SKILL"],
  CONFLICTS_WITH: ["SKILL"],
}
```

这意味着 **profile、preferences、entities、cases、patterns 这 5 类记忆节点永远不能作为关系的终点**——它们被排除在知识图谱之外，只能孤立存在。

**缺陷 B：EDGE_FROM_CONSTRAINT 覆盖不足**

```
from 端仅覆盖 TASK、EVENT、SKILL 3 种
→ profile/preferences/entities/cases/patterns 永远不能作为关系的起点
```

**缺陷 C：3 种 GraphNodeType 与 8 类 MemoryCategory 映射断裂**

8 类记忆中只有 tasks→TASK、skills→SKILL、events→EVENT 有映射关系，其余 5 类（profile、preferences、entities、cases、patterns）没有对应的 GraphNodeType，无法通过图谱 API 建立/查询关系。

**缺陷 D：PATCHES 边未纳入社区统计**

在 `src/graph/community.ts` 中，社区统计的边查询可能遗漏 PATCHES 类型（需确认具体代码）。

## 三、新增边类型设计方案

### 3.1 设计原则

1. **覆盖 8 类记忆的互通需求**：每类记忆至少能建立 1 种关系
2. **语义明确**：边类型名称和方向有清晰含义
3. **与现有 5 种边类型兼容**：不修改现有类型，只新增
4. **LLM 可理解**：extractor prompt 中能清晰描述决策规则

### 3.2 新增 6 种边类型

| 边类型 | from 端 | to 端 | 含义 | 适用场景 |
|--------|---------|-------|------|---------|
| **HAS_PREFERENCE** | TASK / SKILL / EVENT | preferences | 某任务/技能/事件关联的用户偏好 | 用户偏好关联 |
| **BELONGS_TO** | entities / cases / patterns | profile | 某实体/案例/模式属于用户画像的一部分 | 画像关联 |
| **LEARNED_FROM** | patterns | cases | 某模式从某案例中学习得来 | 案例→模式 |
| **EXEMPLIFIES** | cases | skills | 某案例是某技能的具体示例 | 案例关联技能 |
| **RELATED_TO** | 任意类型 | 任意类型 | 两个节点之间存在关联关系（通用兜底） | 跨类别关联 |
| **OBSERVED_IN** | preferences / patterns | events | 某偏好/模式在某事件中观察到 | 事件关联 |

### 3.3 新约束定义

```typescript
export const EDGE_FROM_CONSTRAINT: Record<EdgeType, Set<GraphNodeType>> = {
  // 现有 5 种
  USED_SKILL:     new Set(["TASK"]),
  SOLVED_BY:      new Set(["EVENT", "SKILL"]),
  REQUIRES:       new Set(["SKILL"]),
  PATCHES:        new Set(["SKILL"]),
  CONFLICTS_WITH: new Set(["SKILL"]),
  // 新增 6 种
  HAS_PREFERENCE: new Set(["TASK", "SKILL", "EVENT"]),
  BELONGS_TO:     new Set(["TASK", "SKILL", "EVENT"]),
  LEARNED_FROM:   new Set(["SKILL"]),
  EXEMPLIFIES:    new Set(["SKILL"]),
  RELATED_TO:     new Set(["TASK", "SKILL", "EVENT"]),
  OBSERVED_IN:    new Set(["SKILL", "EVENT"]),
};

export const EDGE_TO_CONSTRAINT: Record<EdgeType, Set<GraphNodeType>> = {
  // 现有 5 种（不变）
  USED_SKILL:     new Set(["SKILL"]),
  SOLVED_BY:      new Set(["SKILL"]),
  REQUIRES:       new Set(["SKILL"]),
  PATCHES:        new Set(["SKILL"]),
  CONFLICTS_WITH: new Set(["SKILL"]),
  // 新增 6 种
  HAS_PREFERENCE: new Set(["TASK", "SKILL", "EVENT"]),
  BELONGS_TO:     new Set(["TASK", "SKILL", "EVENT"]),
  LEARNED_FROM:   new Set(["SKILL"]),
  EXEMPLIFIES:    new Set(["SKILL"]),
  RELATED_TO:     new Set(["TASK", "SKILL", "EVENT"]),
  OBSERVED_IN:    new Set(["SKILL", "EVENT"]),
};
```

### 3.4 方案反思：GraphNodeType 是否需要扩展？

当前方案**不扩展 GraphNodeType**，仍保持 TASK/SKILL/EVENT 三种。原因：

- **MemoryCategory 是分类标签**，不是图节点类型。8 类记忆通过 `category` 字段区分，不需要 8 种 GraphNodeType
- **边约束的是节点类型关系**，不是记忆分类关系。例如 HAS_PREFERENCE 约束"从 TASK/SKILL/EVENT 节点指向 preferences 分类的节点"
- **如果扩展 GraphNodeType 到 8 种**，影响面将翻倍（store.ts 建表、所有查询、extractor prompt、fusion、recall 全部需要修改），属于 v2.0 范畴

**v1.0.0 的最佳策略**：保持 3 种 GraphNodeType 不变，通过新增边类型 + 利用 category 字段实现 8 类记忆的互通。

## 四、重复定义问题

**发现**：`EDGE_FROM_CONSTRAINT` 和 `EDGE_TO_CONSTRAINT` 在两个位置定义：

1. `src/types.ts` 第 38-50 行 — 公开导出
2. `src/extractor/extract.ts` 第 22-37 行 — 私有副本

**风险**：两处必须同步更新，否则会出现行为不一致。

**建议**：
- **方案 A（推荐）**：extract.ts 移除本地定义，改为 `import { EDGE_FROM_CONSTRAINT, EDGE_TO_CONSTRAINT } from "../types"`。风险：需要验证 import 路径无循环依赖
- **方案 B**：两处同步更新，增加代码审查检查项
