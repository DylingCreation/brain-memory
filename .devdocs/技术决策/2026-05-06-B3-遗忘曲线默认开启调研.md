## 一、现状分析

### 1.1 当前衰减机制

`src/decay/engine.ts` 已实现完整的 Weibull 衰减引擎：

| 能力 | 当前状态 |
|------|---------|
| **三层分层衰减** | ✅ core(>0.7) / working(0.4-0.7) / peripheral(<0.4) 不同 beta 和 floor |
| **动态加速** | ✅ dynamic 类型 halfLife / 3 |
| **复合评分** | ✅ recency + frequency + intrinsic 加权 |
| **召回时衰减** | ✅ applyTimeDecay 作为乘性惩罚 |

### 1.2 当前默认配置

```typescript
decay: {
  enabled: false,   // ← 默认关闭
  recencyHalfLifeDays: 30,
  timeDecayHalfLifeDays: 60,
  // ... 其他参数
}
```

### 1.3 衰减公式分析

**scoreDecay**（评分计算）：
```
composite = recencyWeight × weibull(age, halfLife, beta, floor)
          + frequencyWeight × min(1, log10(accessCount+1) / 3)
          + intrinsicWeight × importance
```

**applyTimeDecay**（召回时衰减）：
```
finalScore = rawScore × (0.5 + 0.5 × weibull(age, halfLife, beta, floor))
```

**关键参数影响**：

| 参数 | 默认值 | 含义 | 影响 |
|------|--------|------|------|
| recencyHalfLifeDays | 30 天 | 近期性衰减半衰期 | 30 天后 recency 降到 0.5 |
| timeDecayHalfLifeDays | 60 天 | 召回时衰减半衰期 | 60 天后惩罚系数约 0.75 |
| betaCore | 0.8 | 核心记忆衰减曲线 | <1 表示初期衰减慢、后期加速 |
| coreDecayFloor | 0.9 | 核心记忆最低衰减 | 核心记忆永远不低于 90% |
| peripheralDecayFloor | 0.5 | 外围记忆最低衰减 | 外围记忆最低 50% |

### 1.4 归档需求

当前节点有 `status: "active" | "deprecated"` 字段，但**没有自动归档逻辑**。需要：
1. 在 decay 计算后，对低于阈值的节点标记为 deprecated
2. recall 时排除 deprecated 节点（或通过 composite 过滤）
3. CLI 命令支持 promote/demote 手动调整

## 二、方案设计

### 2.1 默认开启

```typescript
// 修改 DEFAULT_CONFIG
decay: {
  enabled: true,  // ← 从 false 改为 true
  // 其余参数不变
}
```

### 2.2 归档机制

**方案 A：软标记（推荐）**

在 `runMaintenance()` 中增加衰减检查步骤：

```typescript
// 对 composite 低于阈值的节点，标记为 deprecated
const threshold = 0.3;  // configurable
db.prepare(`
  UPDATE bm_nodes SET status = 'deprecated'
  WHERE status = 'active' AND importance < ?
`).run(threshold);
```

**优点**：数据不丢失，可随时恢复。recall 已按 `status='active'` 过滤，自动排除。

**方案 B：压缩表归档**

将 deprecated 节点移到独立的 `bm_nodes_archive` 表，减小主表体积。

**优点**：提升主表查询性能。
**缺点**：增加复杂度，v1.0.0 不需要。

**结论**：v1.0.0 采用方案 A（软标记），方案 B 留作 v2.0 规划。

### 2.3 CLI 命令

新增 3 个 CLI 命令：

| 命令 | 功能 |
|------|------|
| `npm run decay-stats` | 显示各 tier 节点分布、平均 composite 分数 |
| `npm run decay-promote <node-id>` | 手动提升节点为 core（importance → 0.8） |
| `npm run decay-demote <node-id>` | 手动降级节点（importance → 0.2） |

### 2.4 迁移指南

行为变更说明：

```markdown
## 迁移指南：decay 默认开启

### 影响
- v1.0.0 起，decay.enabled 默认从 false 改为 true
- 已有节点不会立即被归档（仅在下次 maintenance 时评估）
- recall 时旧节点的检索分数会受衰减影响

### 如果你想关闭衰减
在 openclaw.json 中设置：
{
  "plugins": {
    "entries": {
      "brain-memory": {
        "config": {
          "decay": { "enabled": false }
        }
      }
    }
  }
}
```

### 2.5 数据库体积影响

衰减开启后，deprecated 节点仍在数据库中（软标记），**数据库体积不会自动减小**。

**体积控制策略**：
1. 短期内：deprecated 节点占比低，体积影响可忽略
2. 长期看：当 deprecated 节点超过 50% 时，可手动执行归档清理
3. v2.0 规划：自动归档到压缩表

## 三、风险评估

| 风险 | 等级 | 预案 |
|------|------|------|
| 衰减导致旧节点 recall 分数降低，用户体验变化 | 中 | applyTimeDecay 的惩罚是乘性的，不是硬删除，旧节点仍可召回 |
| 数据库体积不减小 | 低 | 软标记策略，体积不变但查询过滤 deprecated，不影响性能 |
| 默认行为变更引起兼容性问题 | 中 | 迁移指南 + 用户可手动关闭 |
