## 一、现状分析

### 1.1 allowCrossScope 声明但无实现

`src/scope/isolation.ts` 中 `ScopeFilter` 定义了 `allowCrossScope: boolean`，但 `buildScopeFilterClause()` **完全没有使用这个字段**：

```typescript
// 当前逻辑：只处理 includeScopes 和 excludeScopes
export function buildScopeFilterClause(filter: ScopeFilter): { clause: string; params: any[] } {
  // ... includeScopes → AND 子句
  // ... excludeScopes → != 子句
  // allowCrossScope 被完全忽略！
}
```

### 1.2 当前 recall 的跨范围机制

`ContextEngine.recall()` 使用了两层 fallback：
1. 先按 agentId/workspaceId 查询（includeScopes）
2. 无结果时 fallback 到 `allowCrossScope: true` 的全局查询

**问题**：这不是配置化的共享策略，而是「有/无结果」的硬编码 fallback，无法控制「哪些类别共享」「哪些 Agent 共享」。

### 1.3 数据存储已支持多 Agent

`bm_nodes` 表已有 `scope_agent` 字段，写入时传入 `params.agentId`。数据结构层面已就绪。

### 1.4 共享场景需求

| 场景 | 需求 | 示例 |
|------|------|------|
| PersonalAgent → CodingHelper | profile/preferences 共享 | 用户偏好「简洁回复」，两个 Agent 都遵守 |
| 同一 Agent 不同 Session | 自然共享（已有） | 同一 Agent 的新会话召回历史记忆 |
| 完全隔离 | 不同用户的数据互不可见 | 多租户场景 |

## 二、方案设计

### 2.1 配置化共享策略

在 `BmConfig` 中新增：

```typescript
memorySharing: {
  enabled: boolean;                // default true
  mode: "isolated" | "mixed" | "shared";  // default "mixed"
  /** 允许跨 Agent 共享的记忆类别 */
  sharedCategories: MemoryCategory[];     // default ["profile", "preferences"]
  /** 允许共享的 Agent 列表（空=所有 Agent） */
  allowedAgents: string[];               // default []
}
```

**三种模式**：

| 模式 | 行为 |
|------|------|
| `isolated` | 完全隔离，各 Agent 只能读取自己的记忆 |
| `mixed` | 部分共享，仅 `sharedCategories` 中的类别可跨 Agent |
| `shared` | 完全共享，所有类别跨 Agent 可读 |

### 2.2 实现方案

**步骤 1**：扩展 `buildScopeFilterClause`，支持 `allowCrossScope` + 共享策略

```typescript
// 伪代码
if (filter.allowCrossScope && sharingConfig.mode !== "isolated") {
  // 构建跨 Agent 查询的 includeScopes
  // 按 sharedCategories 过滤（mixed 模式）
  // 按 allowedAgents 过滤（如指定）
}
```

**步骤 2**：在 `ContextEngine.recall()` 中使用配置化策略替代硬编码 fallback

```typescript
// 旧：硬编码 fallback
if (result.nodes.length === 0 && includeScopes.length > 0) {
  result = await this.recaller.recall(query, { allowCrossScope: true });
}

// 新：按配置化策略
const crossScopeFilter = buildCrossScopeFilter(this.config.memorySharing, agentId, workspaceId);
result = await this.recaller.recall(query, crossScopeFilter);
```

**步骤 3**：`ScopeFilter` 增加 `sharedCategories` 字段，store 层在查询时按 category 过滤

### 2.3 冲突解决策略

当多个 Agent 对同一事实有不同记录时：

| 策略 | 说明 |
|------|------|
| 按时间戳优先 | 最新记录覆盖旧记录（适合 preferences 等动态信息） |
| 按 sourceAgent 标记 | 保留所有记录，标记来源 Agent（适合 profile 等静态信息） |
| 按 validatedCount 排序 | recall 排序天然处理（高验证次数优先） |

**推荐**：利用现有的 `validatedCount` + `pagerank` + `updatedAt` 排序，无需额外冲突解决逻辑。recall 排序已经处理了。

## 三、风险评估

| 风险 | 等级 | 预案 |
|------|------|------|
| sharedCategories 过滤影响 recall 性能 | 低 | 在 SQL WHERE 中加 `category IN (...)` 索引友好 |
| 配置默认值不合理导致意外共享 | 中 | 默认 mixed + 仅 profile/preferences，保守策略 |
| 与现有 fallback 机制冲突 | 低 | 新策略完全替代旧 fallback，不保留旧路径 |
