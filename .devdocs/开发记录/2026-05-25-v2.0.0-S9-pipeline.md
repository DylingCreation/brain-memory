## 二、第一次尝试：sed 快速试错（失败）

### 做法

用 `sed` 批量替换 + 快速重写，试图在 20 分钟内完成管线包装。

- 第 1 轮: sed 拆分 store.ts→子模块的函数 → 直接套用到 maintenance 函数
- 第 2 轮: 发现 `detectCommunities` 参数签名不匹配 → sed 替换参数名
- 第 3 轮: 发现返回值类型不匹配 (`DedupResult` ≠ `void`) → 用 `.then(() => {})` 吞掉
- 第 4 轮: sed 替换引入 `async async` 重复关键字、`{` 不匹配 → 手动修复

### 结果

```
27 failed tests (22 introduced by pipeline)

maintenance-coverage:  8 fails — 衰减归档缺失
lite-maintenance:       7 fails — Lite 模式只做了 decay
integration:            3 fails — LLM 社区摘要完全缺失
maintenance:            2 fails — 返回值结构不匹配
hooks:                  1 fail  — performFusion 依赖维护状态
c7-recall-baseline:     1 fail  — 召回依赖图状态
```

### 根因

没有先读完旧函数的每一行。管线只包装了 4 个纯计算步骤（dedup/PR/LPA/decay），遗漏了 3 个副作用步骤：

| 遗漏步骤 | 位置 | 影响 |
|---------|------|------|
| LLM 社区摘要 `summarizeCommunities()` | 全量+增量路径 | 社区摘要永不被生成 |
| decay 归档 `deprecateNode()` | 全部三条路径 | 低价值节点不自动清理 |
| `clearDirty()` | 全部三条路径 | 增量标记不重置，下次维护路径错误 |

### 教训

```
"批量 sed 替换" ≠ "快速推进"
↓
4 轮编译错误 + 22 个测试失败 > 逐行读一遍旧代码的时间
↓
原则: 先调研 → 再设计 → 最后动手
```

## 三、第二次尝试：调研驱动（成功）

### 调研过程

逐行读取 `maintenance.ts` 的 `runFullMaintenancePath()`、`runIncrementalMaintenancePath()`、`runLiteMaintenancePath()`，映射每一步：

```
              全量                         增量                       Lite
              ───                         ───                       ───
1. dedup      dedup(s,c)                  dedup(s,c)                dedup(s,c)
   cache       if merged>0 → invalidate    if merged>0 → invalidate   if merged>0 → invalidate
2. pagerank   computeGlobalPR(s,c)        runIncrementalPR(s,c)      computeGlobalPR(s,c)
3. LPA        detectCommunities(s)         runIncrementalComm(s)      (skip)
4. summaries  summarizeCommunities(comm, llm, embed)                 (skip)
5. decay      逐节点 scoreDecay → if <0.25 && v≤1 → deprecateNode
6. cleanup    clearDirty()                clearDirty()               clearDirty()
```

**关键发现**: LPA 的 `detectCommunities` 只接受 `(storage, maxIter)`，不是 `(storage, cfg)`——之前 4 轮编译失败的根本原因。

### 函数签名分析

| 函数 | 签名 | 返回 |
|------|------|------|
| `dedup` | `(storage: IStorageAdapter, cfg: BmConfig)` | `DedupResult` |
| `computeGlobalPageRank` | `(storage: IStorageAdapter, cfg: BmConfig)` | `GlobalPageRankResult` |
| `runIncrementalPageRank` | `(storage: IStorageAdapter, cfg: BmConfig, threshold?: number)` | `IncrementalPRResult` |
| `detectCommunities` | `(storage: IStorageAdapter, maxIter?: number)` | `CommunityResult` |
| `runIncrementalCommunities` | `(storage: IStorageAdapter, maxIter?: number, threshold?: number)` | `IncrementalCommunityResult` |
| `summarizeCommunities` | `(storage, communities, llm: CompleteFn, embed: EmbedFn)` | `number` |
| `scoreDecay` | `(node: BmNode, cfg: DecayConfig, now?: number)` | `DecayScore` |

### 实现

不再用 `MaintenancePipeline` 委托模式——直接写 `runMaintenance()` 在 `pipeline.ts` 中，保留完整的 `MaintenanceResult` 返回类型与旧结构兼容。

## 四、老板纠正：Clean Break

初始版本将新旧函数共存（`runMaintenance()` 保留原实现 + `runPipelineMaintenance()` 新出口）。

**老板指出**: v2.0.0 已是 breaking 大版本，共存 = 技术债。

**修正**: 
- `maintenance.ts`: 695 行 → 45 行（thin wrapper，委托给 `pipeline.ts`）
- 删除三条独立路径函数（`runFullMaintenancePath` / `runIncrementalMaintenancePath` / `runLiteMaintenancePath`）
- `runMaintenance()` 直接返回 `MaintenanceResult`，所有测试无需改动

## 七、经验提炼

1. **调研先于代码** — 30 分钟的逐行映射省了 20 分钟的 sed 试错 + 15 分钟的回退 + 10 分钟的修复
2. **breaking 版本就该彻底 breaking** — 如果已经做了 Scope 升级这种级别的变更，不要在同一版本留"共存"尾巴
3. **函数签名异构是管线的核心挑战** — `detectCommunities` 的第二个参数是 `maxIter`（number），不是 `cfg`（BmConfig）。不做调研就 sed，必然踩坑
4. **返回值兼容性决定测试影响面** — `MaintenanceResult` 保留 `dedup`/`pagerank`/`community` 等字段，测试零改动是最大的收益
