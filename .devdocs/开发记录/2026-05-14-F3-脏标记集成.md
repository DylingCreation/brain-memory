## 三、操作事故记录

### 事故（What）

在已通过 3 次 `edit` 工具成功修改 `context.ts` 后，尝试用 `awk` 插入 `_expandDirtyMarks()` 方法，导致文件被破坏，产生 100+ 个 TS 编译错误。

### 根因（Why × 3）

```
Why 1：为什么 awk 破坏了文件？
→ awk 的替换模式匹配到了 class 体内的 throw new Error 语句，
  在错误位置插入了方法代码，打断了 class 的方法声明链。

Why 2：为什么 git checkout 后需要重写整个文件？
→ git checkout 回退到了 F-1 改造之前的状态（因为 F-1 的改动只存在于工作树中），
  3 次成功的 edit 改动一并丢失，必须重新应用所有改动。

Why 3：为什么没有在 awk 之后立即验证？
→ 跳过了「分批验证」原则（方法论 §6.3）：每改一批立即跑编译确认。
  直接信任了 awk 输出。
```

### 教训

| 教训 | 方法论对应 |
|------|-----------|
| **awk/sed 等文本操作必须后跟验证步骤** | 通用 §6.3 分批验证 |
| **小步快跑**：每次 edit 后立即 `npm run build` | 同上 |
| **git commit 作为安全网**：如果 F-1 的改动已 commit，checkout 后只需要 apply F-3 改动 | 通用 §5.1 留痕与代码同步 |
| **避免批量文本替换**：优先使用 `write` 重写整个文件 | 本次新增经验 |
