## 问题描述

`ContextEngine` 类（656 行，45+ 方法）直接依赖 15 个模块，承担了存储管理、提取编排、召回编排、融合编排、反思编排、推理编排、工作记忆、维护调度、健康检查、统计、导出/导入、钩子系统等全部职责。任何子系统变更都可能波及，且单元测试几乎无法编写。

## 拆分方案

```
拆分前：ContextEngine (656 行, 45+ 方法)
    │
    │  ┌─ ExtractionService (225 行)            ← 新建
    │  │   processTurn() — 提取管线全流程
    │  │   _expandDirtyMarks()
    │  │
    │  ├─ RecallService (80 行)                 ← 新建
    │  │   recall() — scope 过滤 + 双回退召回
    │  │
    │  ├─ MaintenanceService (36 行)            ← 新建
    │  │   run() — 维护管线 + 能力降级
    │  │
    │  ├─ HealthService (134 行)                ← 新建
    │  │   getStats() / healthCheck() / 简单委托
    │  │
    │  └─ ContextEngine (368 行, -44%)          ← 精简为编排器
    │       constructor — wiring
    │       performFusion / reflectOnSession
    │       performReasoning / workingMemory
    │       export / import / close
```

## 设计要点

1. **WorkingMemory 状态共享**：`ExtractionService.processTurn` 返回 `workingMemory`，ContextEngine 在委托后更新 `this.workingMemory`

2. **类型兼容**：`ProcessTurnParams` / `ProcessTurnResult` 与原 `processTurn` 签名完全一致

3. **依赖注入**：每个 Service 的构造函数仅接收必需的依赖（storage、config、对应组件），无全局状态

4. **对外 API 不变**：所有类型（`EngineStats`、`HealthStatus` 等）通过 `export type {}` 重导出
