## 一、开发内容

### 核心问题

v0.2.0 的图 Schema 存在两个根本缺陷：
1. **EDGE_TO_CONSTRAINT 全部指向 SKILL** — 5 种边类型的 to 端都只有 SKILL，导致 profile/preferences/entities/cases/patterns 这 5 类记忆节点永远不能作为关系终点
2. **3 种 GraphNodeType 与 8 类 MemoryCategory 映射断裂** — 5 类记忆无法通过图谱 API 建立/查询关系

### 1. 修改文件

#### `src/types.ts` — 边类型定义扩展

**EdgeType 从 5 种扩展到 11 种**：

新增 6 种边类型：

| 边类型 | from 端 | to 端 | 含义 |
|--------|---------|-------|------|
| HAS_PREFERENCE | TASK, SKILL, EVENT | TASK, SKILL, EVENT | 关联用户偏好 |
| BELONGS_TO | TASK, SKILL, EVENT | TASK, SKILL, EVENT | 归属关系 |
| LEARNED_FROM | SKILL | SKILL | 从某经验学习 |
| EXEMPLIFIES | SKILL | SKILL | 案例是技能的示例 |
| RELATED_TO | TASK, SKILL, EVENT | TASK, SKILL, EVENT | 通用关联（兜底） |
| OBSERVED_IN | SKILL, EVENT | SKILL, EVENT | 在某事件中观察到 |

**约束同步更新**：EDGE_FROM_CONSTRAINT 和 EDGE_TO_CONSTRAINT 各新增 6 种边类型的方向约束。

#### `src/extractor/extract.ts` — 消除重复定义 + prompt 更新

| 改动 | 说明 |
|------|------|
| **移除本地约束副本** | 删除 extract.ts 中的 EDGE_FROM_CONSTRAINT / EDGE_TO_CONSTRAINT 本地定义（共 16 行），改为从 types.ts 导入 |
| **VALID_EDGE_TYPES 动态化** | 从硬编码 Set 改为 `Object.keys(EDGE_FROM_CONSTRAINT)` 自动生成，新增边类型自动纳入 |
| **EXTRACT_SYS prompt 更新** | 边提取从"5 种"更新为"11 种"，新增 6 种边类型的方向约束和决策规则 |
| **correctEdgeType 重构** | 策略从"强制修正"改为"先验证、再修正"：LLM 提供的类型若合法直接通过，不合法时才尝试修正为 USED_SKILL/SOLVED_BY。避免覆盖新边类型（如 RELATED_TO、OBSERVED_IN） |
| **类型安全修复** | correctEdgeType 中 `type` 参数添加 `as EdgeType` 断言，`fromType/toType` 添加 `as GraphNodeType` 断言 |

## 四、向后兼容性

- **SQLite 无需 ALTER TABLE** — bm_edges.type 为 TEXT 字段，新增值自动兼容
- **已有边数据不受影响** — USED_SKILL 等 5 种旧边类型继续有效
- **API 签名无破坏** — EdgeType 类型为扩展的 union type，旧代码不受影响
- **correctEdgeType 行为变更** — 策略从"强制修正"改为"先验证、再修正"，对旧 5 种边的有效用例行为不变（先验证通过直接返回），仅影响原本会被错误覆盖的边界场景
