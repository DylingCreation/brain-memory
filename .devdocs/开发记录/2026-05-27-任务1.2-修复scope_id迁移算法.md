# 任务 1.2：修复 scope_id 迁移算法不一致

**来源**: M3-P0-1  
**执行日期**: 2026-05-25  
**执行人**: AI  
**状态**: ✅ 已完成

---

## 问题描述

`migrate.ts` 和 `isolation.ts` 使用两种不同的算法生成 scope_id，导致迁移后的旧节点在 scope 过滤时永远无法被匹配到。

### 算法差异

| 位置 | 算法 | 示例输出（platform=qqbot） |
|------|------|--------------------------|
| `migrate.ts:117` (SQL, 旧) | `lower(hex(CONCAT(...)))` — ASCII hex 编码 | `7171626f747c2a7c...` (52 chars) |
| `isolation.ts:123` (JS, 规范) | `sha256(...).hex.slice(0,16)` — 哈希摘要 | `9f0c47aba5acb0cf` (16 chars) |

### 影响

- 迁移产生的 scope_id 与 `computeScopeId()` 新生成的 scope_id **完全不同**
- scope 过滤查询（`WHERE scope_id = ?`）无法匹配旧节点
- 导致 scope 隔离机制对已迁移数据**完全失效**

### 根因

SQLite 没有内置 sha256 函数，初版迁移选择了 `hex(CONCAT(...))` 作为简化替代，但忽略了与 JS 端算法的兼容性。

---

## 修复内容

### 方案：JS 逐行计算 → 事务批量更新

将 SQL 的 `UPDATE ... SET scope_id = lower(hex(CONCAT(...)))` 替换为 JS 层逐行调用 `createHash('sha256')`，通过事务批量写入。

#### 修改文件：`src/store/migrate.ts`

**1. 新增 import**
```typescript
import { createHash } from 'crypto';
```

**2. 替换 SQL UPDATE 为 JS 计算**

```typescript
// 修复后：JS 逐行计算 sha256 scope_id
const computeScopeIdV2 = (platform, workspace, agent, user, chat, thread) => {
  const parts = [platform ?? '*', workspace ?? '*', agent ?? '*', user ?? '*', chat ?? '*', thread ?? '*'];
  return createHash('sha256').update(parts.join('|')).digest('hex').slice(0, 16);
};

const rows = db.prepare(`
  SELECT id, scope_platform, scope_workspace, scope_agent,
         scope_user, scope_chat, scope_thread
  FROM bm_nodes WHERE scope_id IS NULL OR scope_id = ''
`).all();

if (rows.length > 0) {
  const updateStmt = db.prepare('UPDATE bm_nodes SET scope_id = ? WHERE id = ?');
  db.transaction(() => {
    for (const row of rows) {
      const scopeId = computeScopeIdV2(row.scope_platform, ...);
      updateStmt.run(scopeId, row.id);
    }
  })();
}
```

**3. 幂等性**：查询条件 `WHERE scope_id IS NULL OR scope_id = ''` 确保已计算过的行不会被重复更新。

## 影响分析

- **风险**：低。仅影响 scope_id 生成逻辑，不改变数据库 schema
- **数据兼容**：幂等设计 — 已运行过旧迁移的数据库，重复运行迁移会**覆盖旧 scope_id 为正确值**。已有 null scope_id 的行会被更新，已有正确值的行不会重复处理
- **副作用**：旧迁移数据被修复后，scope 过滤查询将正确返回这些节点（之前被静默排除）
- **性能**：逐行 UPDATE 在事务中执行，万级节点迁移耗时 < 1 秒
