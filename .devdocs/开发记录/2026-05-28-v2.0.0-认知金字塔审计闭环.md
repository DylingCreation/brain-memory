## 二、第一阶段：测试修复（5 项 bug 修复）

### 2.1 source 列迁移（`src/store/migrate.ts`）

**问题**：`ContextEngine` 构造函数中 `storage.getStats()` 查询 `bm_nodes WHERE source='user'`，但旧版 SQLite 数据库（v2.0 之前创建的）的 `bm_nodes` 表缺少 `source` 列。`hooks.test.ts` 和 `integration.test.ts` 触发此错误。

**根因**：`source` 列在 `db.ts` SCHEMA 中已定义（`TEXT NOT NULL DEFAULT 'user'`），但现有数据库通过 `CREATE TABLE IF NOT EXISTS` 创建，表已存在时不会新增列。迁移系统未包含 source 列的补加逻辑。

**修复**（`src/store/migrate.ts` 第 105-106 行）：
```typescript
// 在 migrateToV2_ScopeUpgrade() 中，紧接 scope 列迁移之前：
// 0. source 列补加（幂等：列已存在则跳过。v2.0.1 之前创建的数据库可能缺此列）
try { db.exec("ALTER TABLE bm_nodes ADD COLUMN source TEXT NOT NULL DEFAULT 'user' CHECK(source IN ('user', 'assistant', 'manual'))"); } catch { /* 列已存在 */ }
```

**验证**：`hooks.test.ts` 的 `beforeFusion hook` 测试不再抛出 `no such column: source`。全量测试中 `ContextEngine.performFusion` 正常执行。

### 2.2 getAllCommunities 测试调用修复（`test/integration/integration.test.ts`）

**问题**：`integration.test.ts` 第 417 行和第 510 行调用 `assembleContext(db, ...)` 传入的是 `DatabaseSyncInstance`（原始 SQLite 连接），而非 `IStorageAdapter`。`assembleContext` 内部调用 `storage.getAllCommunities()`，但 `DatabaseSyncInstance` 没有此方法。

**根因**：模板字符串在测试代码中的 `beforeEach` 块同时创建了 `storage`（IStorageAdapter）和 `db`（通过 `storage.getDb()` 获取的原始连接）。测试编写者误用了 `db` 变量。

**修复**：

// 修改后：

```

### 2.3 c7-recall-baseline getRawDb（`src/engine/context.ts` + `test/performance/c7-recall-baseline.test.ts`）

**问题**：`c7-recall-baseline.test.ts` 第 59 行调用 `engine.getDb()`，但 `ContextEngine` 没有此方法。测试需要通过原始 SQL 批量插入 200 个节点做性能基准——走 `upsertNode` API 太慢。

**修复**：在 `ContextEngine` 新增 `@internal` 方法供测试/基准场景使用：

/** @internal Get the raw database instance (for testing/benchmarking only). */
getRawDb(): ReturnType<typeof import('../store/sqlite-adapter').SQLiteStorageAdapter.prototype.getDb> {

}
// 测试中 engine.getDb() → engine.getRawDb()
```

### 2.4 SQLite 锁冲突（`test/helpers.ts`）

**问题**：vitest 配置 `pool: 'forks'`（每个测试文件独立进程），`createTestStorage()` 使用 `join(tmpdir(), 'brain-memory-test-{timestamp}-{counter}.db')` 创建文件数据库。多个测试文件并发访问同一 tmpdir 路径时，`DatabaseSync` 开启 WAL 模式也可能触发 `SQLITE_BUSY`。多文件同时初始化/写入同一文件时锁冲突概率增加。

**修复**：全部测试改用 `:memory:` 数据库——每个 `SQLiteStorageAdapter` 实例拥有完全隔离的内存数据库。
```typescript
// 修改前：
export function createTestStorage(): IStorageAdapter {
  _testCounter++;
  const dbPath = join(tmpdir(), `brain-memory-test-${Date.now()}-${_testCounter}.db`);
  const storage = new SQLiteStorageAdapter(dbPath);
  storage.initialize();
  return storage;
}

// 修改后：
export function createTestStorage(): IStorageAdapter {
  const storage = new SQLiteStorageAdapter(":memory:");
  storage.initialize();
  return storage;
}
```
同时删除了不再使用的 `_testCounter` 变量，简化了 `cleanupTestDb()` 函数。

**附带修复**（`test/unit/hooks.test.ts`）：`hooks.test.ts` 使用 `new ContextEngine({ ...DEFAULT_CONFIG })`，而 `DEFAULT_CONFIG.dbPath = '~/.openclaw/brain-memory.db'`——打开真实生产数据库。修复为：
```typescript
const TEST_CONFIG = { ...DEFAULT_CONFIG, dbPath: ":memory:" };
// 所有 new ContextEngine({ ...DEFAULT_CONFIG }) → new ContextEngine({ ...TEST_CONFIG })
```

### 2.5 recaller.test.ts 导入路径（`test/integration/recaller.test.ts`）

### 3.2 D9：路径感知融合（`src/recaller/recall.ts`）

**设计构想**：三路候选（SQLite关键词 + LanceDB向量 + USER.md/MEMORY.md）→ 合并去重 → "关键词和语义都命中就融合分数"。

**实际代码**：所有种子统一进入单次 PPR 排序，路径来源信息在合并后丢失。无法区分"仅 FTS5 命中"和"FTS5 + LanceDB 双命中"的节点。

**修复**（`Recaller.recall()` 方法）：
```typescript
// 新增 pathOrigin Map：bitmap 标记每个种子节点的来源路径
const pathOrigin = new Map<string, number>();
// 1 = 精确路径 (preciseSeeds)
// 2 = 泛化路径 (generalizedSeeds)
// 4 = 语义路径 (semanticSeeds / LanceDB ISearchIndex)
// 8 = 外部路径 (externalNodes / USER.md+MEMORY.md)

// 排序时：多路命中节点 ×1.2 加权
const MULTI_PATH_BOOST = 1.2;
const pathCount = [1, 2, 4, 8].filter(b => (originBits & b) !== 0).length;
const finalScore = pathCount >= 2 ? baseScore * MULTI_PATH_BOOST : baseScore;
```

## 四、第三阶段：改进建议（P2/P3）

### 4.1 D8：召回前置信息量判断（3 文件）

**设计构想**：用户消息先判断是否值得查记忆，"好的""继续""嗯"等低信息量内容跳过召回。

**修复**（`src/noise/filter.ts`）：
```typescript
// 新增 shouldRecall() 函数
const RECALL_SKIP_PATTERNS = [
  /^(好的|收到|嗯嗯|哦哦|哈哈|呵呵|行|可以|OK|ok|okay|k+|yes|no|yep|nope|sure|right|fine|got it|gotcha|继续|go on|next|然后|然后呢|之后|之后呢)$/i,
  /^(嗯|哦|啊|哈|额|呃|咦|哟|嗨|嘿)$/,
];
const RECALL_SKIP_EMOJI = new Set(['👍','👌','✅','❌','❤️','😂','🙏','🔥','💯']);

export function shouldRecall(text: string): boolean {
  const trimmed = text.trim();
  if (trimmed.length === 0) return false;
  if (trimmed.length < 3) return false;
  for (const pattern of RECALL_SKIP_PATTERNS) if (pattern.test(trimmed)) return false;
  if ([...trimmed].every(c => RECALL_SKIP_EMOJI.has(c))) return false;
  if (GREETING_RE.test(trimmed) && trimmed.length < 15) return false;
  return true;
}
```

**接入点**：
1. `src/engine/recall-service.ts` — `RecallService.recall()` 入口：`if (!shouldRecall(query)) return { nodes: [], edges: [], tokenEstimate: 0 };`
2. `src/plugin/core.ts` — `PluginCore.getMemoryContext()` 入口：`if (!shouldRecall(queryText)) { logger.debug(...); return null; }`

### 4.2 R2：缓存失效策略改进（`src/recaller/cache.ts`）

**原实现**：`RecallCache.isValid()` 仅检查 `storage.getDirtyNodes().size === 0`。若节点内容变更但未走 markDirty 路径，缓存返回过期数据。

**修复**：
```typescript
// 新增 _contentHash 字段 + _computeContentHash() 方法
private _contentHash: string | null = null;

isValid(storage: IStorageAdapter): boolean {
  if (storage.getDirtyNodes().size > 0) { this._contentHash = null; return false; }
  const currentHash = this._computeContentHash(storage);
  if (this._contentHash === null) { this._contentHash = currentHash; return true; }
  if (this._contentHash !== currentHash) { this._contentHash = currentHash; this.invalidate(); return false; }
  return true;
}

private _computeContentHash(storage: IStorageAdapter): string {
  const nodes = storage.findAllActive();
  const key = nodes.map(n => `${n.id}:${n.updatedAt}`).sort().join(',');
  return createHash('md5').update(key).digest('hex');
}
```
- 无脏节点时 → 计算内容 hash 比对 → hash 不同则 invalidate 全部缓存
- 内容 hash = 所有活跃节点 `{id}:{updatedAt}` 排序后拼接的 MD5
