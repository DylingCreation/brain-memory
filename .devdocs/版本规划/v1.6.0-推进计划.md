## 一、版本定位与核心目标

### 定位

v1.5.0 完成了「Lite 模式 + 项目身份」。
v1.6.0 是**工程夯实 + 性能起步版本**，按三层优先级推进：C（工程收尾）→ A（性能深化）→ B（功能增强）。

### 核心目标（不可调整）

| 编号 | 核心目标 | 验收底线 |
|------|---------|---------|
| C-目标 | 工程卫生达标 | lint 0 error、npm audit 0 漏洞、核心模块覆盖率 ≥ 70% |
| A-目标 | 性能基座升级 | 查询缓存可用、10k 节点性能验证通过 |
| B-目标 | Small 模式可用 | `mode: "small"` 存在、精简 prompt 生效、边界验证矩阵通过 |

### 可调整范围

在核心目标不变的前提下：
- **批次顺序**可调整（如遇阻塞先绕路）
- **批次预估天数**可调整（实际工作量偏差 ≤ 50% 属正常）
- **L2 等级功能**可降级发布（如 LanceDB 生产化遇坑，降级为 L2 后仍可发布）
- **覆盖率目标**可微调（核心模块 ≥ 70% 为底线，整体 78%+ 为理想值）

## 六、调整机制

### 6.1 可调项 vs 不可调项

| 类型 | 可调 | 不可调 |
|------|------|--------|
| 核心目标 | — | C/A/B 三大目标底线 |
| 批次顺序 | ✅ 可跳过、可前置、可并行 | — |
| 批次预估 | ✅ 实际工作量偏差 ≤ 50% | — |
| L2 等级功能 | ✅ 可降级为 L3 发布 | — |
| 覆盖率底线 | ⚠️ 整体 78% 可微调 | 核心模块 ≥ 70% 不可低于 |
| 版本发布日期 | ✅ 可提前/延后 | — |

### 6.2 调整触发条件

| 触发条件 | 调整方式 |
|----------|---------|
| 某批次实际工作量 > 预估 × 2 | 拆分该批次，或延至后续版本 |
| 某批次遇技术阻塞 | 先跳过，继续下一批次 |
| lint 收敛发现工作量超预期 | 只做 P0（加 `_` 前缀），「其他」类延后 |
| 依赖升级失败 | 标记原因，跳过 |
| 新发现的 blocker（如 bug） | P0 优先级插入 |

### 6.3 调整流程

1. 发现需要调整时，在推进计划中标注「已调整」
2. 记录调整原因、调整内容、对新计划的影响
3. 向老板汇报调整（简洁版：What + Why + How）
4. 确认后即可继续推进
