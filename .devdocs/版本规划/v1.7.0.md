## 三、功能详述

### F-1 测试分层方案

**问题**：性能基准测试和单元/集成测试混在 `vitest run` 的同一个 fork pool 里跑。4 个 fork 进程同时跑 10K 节点图运算 → CPU 竞争 → 时序不准、断言抖动、LanceDB 竞态。

**方案**：

```
npm test                          npm run test:perf
(maxForks: 4)                     (maxForks: 1, 独立 config)
├─ unit/*                          ├─ perf-tiered-benchmark
├─ integration/*                   ├─ perf-1k-benchmark
└─ (排除 perf + lancedb)           ├─ perf-content-benchmark
                                   └─ (lancedb-poc 条件跳过)
```

**实施项**：
1. 新建 `vitest.perf.config.ts`，`maxForks: 1`，`testTimeout: 60000`，`include: ['test/perf-*.test.ts']`
2. 修改 `vitest.config.ts`，`exclude` 新增 `test/perf-*.test.ts`、`test/lancedb-*.test.ts`
3. `package.json` 新增 `test:perf`、`test:all` scripts
4. `test/lancedb-poc.test.ts` 向量测试 suite 加 `describe.skipIf` 条件跳过（fork 模式下跳过，单进程仍可跑）
5. METHODOLOGY.md §2 发布检查清单更新

**验收标准**：
- `npm test`：排除 perf，0 failed
- `npm run test:perf`：perf 16 tests，maxForks:1，0 failed
- `npm run test:all`：test + test:perf 均通过
- `npx vitest run test/lancedb-poc.test.ts`：单文件跑全过（14 passed, 1 skipped）

### F-2 test/ lint 收敛

**现状**：`npm run lint` 只跑 `src/**/*.ts`。test/ 目录 3466 errors（3307 可 `--fix` 自动修复，159 需手动），几乎全为 `quotes` 规则（测试文件习惯双引号）。

**方案**：F-8 先放宽规则后，对剩余错误分类处理：

| 错误类型 | 数量（预计） | 处理方式 |
|---------|-------------|---------|
| `quotes` | ~3300 | F-8 放宽规则 → 不再报错 |
| `no-unused-vars` | ~80 | 分析后删除或加 `_` 前缀 |
| `prefer-const` | ~50 | `--fix` 自动修 |
| `no-explicit-any` | ~20 | 按 §5.7 三层模式收敛 |
| 其他 | ~10 | 逐项手动修 |

**验收标准**：`npx eslint 'test/**/*.ts'` 0 errors, 0 warnings

### F-7 Small 模式边界矩阵补全

> **不包含**输出质量评测、多 model 兼容性验证、真实场景端到端测试。
> v1.7.0 完成后 Small 模式评级为 **L2 功能可用**，升至 L1 生产就绪需独立版本的质量评测专项。

**现状**：METHODOLOGY §5.12 定义了 Full/Small/Lite 三层验证矩阵，但测试只有 `small-mode.test.ts`（提示词大小对比 + 路由），未逐单元格覆盖行为正确性。

### F-8 test/ lint 规则差异化

**问题**：test 文件习惯双引号，但全局 `quotes: ["error", "single"]` 强制单引号，导致 3300+ errors。

**方案**：ESLint flat config 追加 test 目录规则覆盖：

```javascript
// eslint.config.mjs
{
  files: ["test/**/*.ts"],
  rules: {
    quotes: "off",              // test 不强制引号风格
    "@typescript-eslint/no-explicit-any": "warn",  // 测试放宽
  },
}
```

**验收标准**：
- test/ lint 中 `quotes` 不再报错
- src/ 规则不受影响
- `npm run lint` 和 `npx eslint 'test/**/*.ts'` 均表现正确

## 附录 A：Small 模式生产就绪度评估

> 本节厘清 Small 模式在 v1.7.0 前后的能力层级，避免将「代码正确」等同于「生产可用」。

### A.1 评级定义（依据工程协作手册 §4.1）

| 等级 | 含义 | Small 模式当前 |
|------|------|--------------|
| L4 开发中 | 功能未完成 | — |
| L3 实验性 | 原型完成，测试不全 | — |
| **L2 功能可用** | 功能可用、逻辑正确，但输出质量/文档未充分验证 | **← v1.7.0 F-7 完成后** |
| L1 生产就绪 | 全维度达标、质量评测完成 | 待 v1.8.0+ |

### A.2 v1.7.0 F-7 覆盖了什么

| 维度 | 内容 | 验证方式 |
|------|------|---------|
| 路由正确性 | processTurn / fusion / reflection / reasoning / maintenance 在 Small 模式下的代码路径正确 | 单元测试（F-7 新增） |
| 跳过逻辑 | Small 模式下 fusion / reflection / reasoning 被正确跳过 | 断言测试 |
| 维护路径 | Small 模式下 maintenance 走全量路径（与 Full 相同） | 已有 `lite-maintenance.test.ts` |
| Lite 不调 LLM | Lite 模式 processTurn 不调用 LLM | 已有测试 |
| JSON 容错 | `smallJsonRepair()` / `safeJsonParse()` 处理小模型 malformed JSON | 已有 `json-enhanced.test.ts` |
| 提示词大小 | Small prompt ~200 tokens（4-15x 压缩） | 已有 `small-mode.test.ts` |

### A.3 生产就绪还缺什么

| 缺失项 | 影响 | 建议版本 |
|--------|------|---------|
| **输出质量对比评测** | 不知道 Small prompt 下提取/融合质量相对 Full 下降多少 | v1.8.0 |
| **多 model 兼容性验证** | 仅设计假设适用于小模型，未针对 qwen3.5:9b / llama3.2:3b 等实测 | v1.8.0 |

| **降级路径验证** | Small 模式 + LLM 不可用时行为是否优雅 | v1.8.0 |
