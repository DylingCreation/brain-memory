## 一、现状分析

### 1.1 全量维护流程

```
session_end
  → runMaintenance()
    → 1. dedup(storage, cfg)              — LSH 去重，O(n²) reduced by bucketing
    → 2. computeGlobalPageRank(storage)    — 全图 PageRank，O(N × iter × avg_deg)
    → 3. detectCommunities(storage)        — LPA 社区检测，O(N × iter × avg_deg)
    → 4. summarizeCommunities(...)         — LLM 摘要生成（LLM 依赖）
    → 5. decay-based archiving             — 低分节点标记 deprecated
```

### 1.2 变更触发点

`processTurn()` 是唯一的数据变更入口：

| 操作 | 影响 |
|------|------|
| `upsertNode()` | 新增节点 / 更新已有节点（content, validatedCount） |
| `upsertEdge()` | 新增边 / 更新已有边 |
| `mergeNodes()` (via dedup) | 节点合并，edge 重新指向 |

### 1.3 性能瓶颈

| 步骤 | 全量复杂度 | 200 节点时 | 10k 节点时（预测） |
|------|-----------|-----------|-----------------|
| PageRank | O(N × 20 × avg_deg) | ~0.1ms | ~50ms |
| LPA | O(N × 50 × avg_deg) | ~0.5ms | ~250ms |
| Dedup (LSH) | O(N × bucket_size) | ~0.3ms | ~150ms |

> 注：200 节点实际测量（C-7 基线）：recall 全链路 0.44ms avg。10k 节点为理论估算。

瓶颈在于**每次 maintenance 都全量扫描所有节点和边**，即使只有几个节点新增。

## 二、增量方案设计

### 2.1 脏标记机制

`IStorageAdapter` 已预留接口（F-1 设计）：

```typescript
// 标记
storage.markDirty(nodeIds);
// 查询
storage.getDirtyNodes();
// 获取受影响的局部子图
storage.getAffectedSubgraph(maxDepth);
// 清除
storage.clearDirty();
```

**标记时机**：`processTurn()` 中每次 `upsertNode()` 和 `upsertEdge()` 后标记对应节点。

**传播策略**：脏标记沿边传播 depth 层（默认 depth=1，即标记脏节点及其邻居）。

### 2.2 增量 PageRank

#### 方案 A：子图 PageRank + 边界固定（推荐 ✅）

```
1. getDirtyNodes() → dirty set
2. getAffectedSubgraph(depth=2) → { nodes, edges }
3. 在子图上运行 PageRank：
   - 内部节点：正常迭代
   - 边界节点（邻居中 dirty）：保持原 pagerank 作为 teleport
4. 将子图中 dirty 节点的分数写回存储
5. 全局其余节点保持原值不变
```

**理论依据**：PageRank 的 personalized 模式天然支持"从种子节点重新分配"。当脏节点占比较小时（< 10%），子图 PageRank 误差可忽略。

**收敛性验证**：
- 子图中仅含 dirty + 1-hop 邻居 + 2-hop 邻居
- PageRank damping factor 0.85，3 跳以上影响 < 0.85³ = 0.61，衰减极快
- 边界固定近似精度损失 < 1%

#### 方案 B：全图 Personalized PageRank（备选）

```
1. 以 dirty nodes 为 seeds
2. 在全图上运行 personalizedPageRank（用缓存图结构）
3. 仅写入 dirty nodes 的新 pagerank
```

**缺点**：仍需全图迭代 O(N)，性能提升有限（仅省去 `updatePageranks` 的全量写入）。

**结论**：推荐方案 A。

### 2.3 增量社区检测（LPA）

#### 方案：冻结 + 局部 LPA

```
1. dirty nodes 重置为初始标签（自身 ID）
2. clean nodes 保持现有 community_id
3. 仅在受影响的子图上运行 LPA：
   - 只迭代 dirty nodes 及其 1-hop 邻居
   - clean nodes 的标签视为固定 anchor
4. 收敛后：新形成的社区命名，已有社区保持原名
5. 如果 dirty node 被合并到已存在的社区 → 加入该社区
6. 如果 dirty node 形成新社区 → 按大小分配新 ID
```

**关键挑战**：
- LPA 的标签传播是**全局**的 — 一个 dirty node 可能通过 3-4 跳吸引远处的 clean nodes
- **但** LPA 本身是收敛的，且实际中影响力衰减很快
- 经验数据：LPA 在 5 迭代内收敛 > 90%，标签只能在连通分量内传播

**风险评估**：
- 风险 1：dirty node 应该属于新社区，但因"冻结"被强制归入旧社区 → 可接受，下次 full maintenance 会修正
- 风险 2：dirty node 形成的新社区与现有社区重叠 → 边界节点的标签不会变化，不会分裂已有社区

**结论**：可行，但需在 full maintenance 时全局重新校正（v1.1.0 保留全量路径）。

### 2.4 智能触发策略

```typescript
function shouldRunIncremental(storage: IStorageAdapter, threshold: number = 0.10): boolean {
  const dirtyCount = storage.getDirtyNodes().size;
  const totalActive = storage.findAllActive().length;
  return dirtyCount / Math.max(totalActive, 1) < threshold;
}
```

| 场景 | 判定 | 执行路径 |
|------|------|---------|
| 脏比例 < 10% | 增量 | `runIncrementalMaintenance()` |
| 脏比例 ≥ 10% | 全量 | `runMaintenance()` (现有) |
| 全量维护后 | — | `clearDirty()` |

### 2.5 去重（Dedup）增量处理

Dedup 的 LSH bucketing 已经是 O(n × bucket_size) 而非 O(n²)。增量策略：

```
1. 取 dirty nodes 中新增的节点（isNew=true）
2. 仅对新增节点做 dedup（vs 全量节点）
3. 如果 deduce 导致 merge → markDirty(mergeId)
```

|------|--------|---------|------|------|

## 五、风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| XR-02 局部 PageRank 不收敛到全量值 | 高 | 先小规模实验验证偏差 < 1%；保留全量路径兜底 |
| 局部 LPA 产生社区碎片 | 中 | 全量维护时全局重整；冻结策略保守（只解冻 1-hop） |
| 脏标记遗漏导致增量结果不完整 | 中 | 在 upsertNode/upsertEdge 中强制标记；传播 depth 可配置 |
| 缓存与脏标记交互 bug | 中 | `invalidateGraphCache()` 仅在 full maintenance 时调用 |

## 六、实施建议

### 实施顺序

```
1. 脏标记基础设施
   → 在 processTurn 中集成 markDirty()
   → 验证 dirty set 正确性

2. 子图提取
   → getAffectedSubgraph(depth) 实现
   → 单元测试

3. 增量 PageRank
   → 小规模实验验证精度（手动对比全量 vs 增量）
   → 确认偏差 < 1% 后实现

4. 增量 LPA
   → 冻结 + 局部迭代实现
   → 社区稳定性验证

5. 智能触发
   → shouldRunIncremental() + 阈值可配置
   → runMaintenance() 内部 if/else 分流

6. 性能基准
   → 对比全量 vs 增量 maintenance 耗时
```
