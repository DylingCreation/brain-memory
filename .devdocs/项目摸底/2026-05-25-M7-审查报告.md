### 🟠 P1 — 高危级

#### P1-1: `ContextEngine` 构造函数硬编码 `SQLiteStorageAdapter` — LanceDB 后端配置无效

**What**: `context.ts:67-68`：
```typescript
this.storage = new SQLiteStorageAdapter(config.dbPath);
this.storage.initialize();
```

尽管 `BmConfig` 定义了 `storage: StorageBackend` 字段（可选 `'sqlite' | 'lancedb'`），且 `openclaw.plugin.json` 的 configSchema 也提供了 `"storage": { "enum": ["sqlite", "lancedb"] }`，但 `ContextEngine` **从不读取** `config.storage` 字段。

**Why 这是问题**: 
- 用户在配置文件中设置 `"storage": "lancedb"` 后，引擎仍然创建 SQLite 适配器
- 用户静默获得与预期不同的存储后端，且没有任何警告
- M3 发现的 LanceDB 适配器 stub 问题因此被掩盖——因为 LanceDB 路径从未被真正使用

**Root Cause**: `ContextEngine` 在 v1.1.0 引入 `IStorageAdapter` 抽象时，选择了"先统一用 SQLiteAdapter，后续再支持多后端"的策略。但该 TODO 从未被完成。

**Recommendation**: 
```typescript
if (config.storage === 'lancedb') {
  this.storage = new LanceDBStorageAdapter(config.dbPath);
} else {
  this.storage = new SQLiteStorageAdapter(config.dbPath);
}
```
并在文档中明确 LanceDB 的功能限制。

#### P1-2: `matchExportScope` 是 scope 匹配逻辑的第四次独立实现

**What**: `context.ts:529-537` 的 `matchExportScope` 函数：
```typescript
if (scope.platform && node.scopePlatform !== scope.platform) return false;
if (scope.workspace && node.scopeWorkspace !== scope.workspace) return false;
// ... 6 个字段的等值匹配
```

**Why 这是问题**: 这是代码库中 scope 匹配逻辑的**第四次**独立实现：

| # | 位置 | 函数 | 匹配语义 |
|---|------|------|----------|
| 1 | `scope/isolation.ts` | `scopeMatchV2()` | 前缀匹配，NULL=通配 |
| 2 | `lancedb-adapter.ts` | `filterByScopeV2()` | 调用 scopeMatchV2 |
| 3 | `recaller/recall.ts` | `matchesScope()` | 等值匹配 |
| 4 | **`engine/context.ts`** | **`matchExportScope()`** | 等值匹配 |

M1-M6 反复发现的 scope 分散问题在此再次出现。

**Recommendation**: 统一使用 `scopeMatchV2()`，删除 `matchExportScope` 并替换为调用 `scopeMatchV2`。

#### P1-3: `performReasoning` 将全部活跃节点传给 LLM — 可能超出上下文窗口

**What**: `context.ts:377-380`：
```typescript
const nodes = this.storage.findAllActive();  // ← 所有活跃节点
const edges = this.storage.findAllEdges();   // ← 所有边
const reasoningResult = await runReasoning(createCompleteFn(this.config.llm)!, nodes, edges, query, this.config);
```

**Why 这是问题**: 
- 当活跃节点达到数千个时，将所有节点和边的文本传给 LLM 会超出模型的上下文窗口（即使是 128K token 的模型）
- 推理引擎本应基于**召回结果**（recall 返回的相关节点）而非全量节点进行推理
- `runReasoning` 的设计注释说 "based on a set of related knowledge nodes"，但 `ContextEngine.performReasoning` 传入了不相关的全量数据

**Root Cause**: `performReasoning` 是一个独立于 `recall()` 的方法，不接收召回结果。它被设计为"对整个图谱做推理"，但实现中忽略了图的大小。

**Recommendation**: 要么限制节点数量（如 Top-100 by pagerank），要么让 `performReasoning` 接受召回结果作为输入。

---

#### P1-4: `extractCommitments` 朴素关键词匹配产生误报

**What**: `working-memory/manager.ts:166-175`：
```typescript
if (lowerText.includes(keyword)) {
  // ... 提取包含关键词的句子
}

// 关键词列表包含：
'i will', 'i\'ll', 'i would', 'should', 'recommend', ...
```

**Why 这是问题**: 
- "I will **not** do that" → 被提取为承诺（实际是拒绝）
- "You **should** use a different approach" → 被提取为 AI 的承诺（实际是建议用户）
- "Let me **check** if that's possible" → 被提取为承诺（实际是条件性表达）
- 完全没有 NLP/否定检测

**Recommendation**: 至少添加否定词检测（`won't`, `not`, `can't`, `don't`），或使用 LLM 进行承诺提取。

### 🟡 P2 — 中等级

#### P2-1: `performFusion` 每次调用创建新的 `createEmbedFn`

**What**: `context.ts:364`：
```typescript
const result = await runFusion(this.storage, this.config, ..., createEmbedFn(this.config.embedding), ...);
```

**Why 这是问题**: 
- `createEmbedFn` 可能建立 HTTP 连接或初始化客户端对象
- 每次融合调用都创建新实例，无法复用连接池
- `ContextEngine` 构造函数中已经创建了 embed 函数但没有保存为实例字段

**Recommendation**: 将 `embedFn` 保存为 `this.embedFn` 实例变量，所有方法共用。

---

#### P2-2: `getStats` 硬编码 `sessionCount: 0`

**What**: `context.ts:481`：
```typescript
return {
  nodeCount: stats.totalNodes, edgeCount: stats.totalEdges, sessionCount: 0,  // ← 硬编码
```

**Why 这是问题**: 
- `sessionCount` 在所有情况下返回 0，即使数据库中有数千条不同 session 的消息
- 这是一个已知未实现的功能（应通过 `SELECT COUNT(DISTINCT session_id) FROM bm_messages` 查询）
- 暴露给 UI 层的统计信息不完整

**Recommendation**: 从 `bm_messages` 表查询真实的 session 计数。

---

#### P2-3: `assembleContext` 中不可达的 `!xml` 检查

**What**: `assemble.ts:125`：
```typescript
if (!xml) return { xml: null, systemPrompt: '', tokens: 0, episodicXml: '', episodicTokens: 0 };
```

**Why 这是问题**: 
- 该检查上面，`recallStrategy === 'off'` 的情况已经通过 `return { xml: null, ... }` 提前返回了（第 119-120 行）
- 在此之后，`xml` 变量**永远不为空字符串**
- 这是一个死代码块，永远无法到达

**Recommendation**: 删除此不可达代码。

---

#### P2-4: Embedding 缓存使用模块级全局变量，多实例共享

**What**: `embed.ts:35-37`：
```typescript
const embedCache = new Map<string, CacheEntry>();
let cacheHits = 0;
let cacheMisses = 0;
```

**Why 这是问题**: 
- 两个使用不同 embedding 模型（如 text-embedding-3-small vs bge-m3）的 `ContextEngine` 实例共享同一个缓存
- 不同模型的向量不可互换——从缓存返回的向量可能来自错误的模型
- 缓存键仅使用文本 hash，不包含模型名

**Recommendation**: 将缓存移到实例级别（从 `createEmbedFn` 闭包中返回），缓存键加入 model 名称。

### 🟢 P3 — 低等级

#### P3-1: LLM 四个端点函数存在重复样板代码

四个函数（`ollamaNativeComplete`, `dashscopeComplete`, `openaiComplete`, `anthropicComplete`）共享相同的结构：
```typescript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 60000);
try { ... } finally { clearTimeout(timeoutId); }
```
约 25 行重复样板 × 4 = 100 行可提取。

#### P3-2: `processTurn` source 检测未处理混合消息场景

```typescript
if (assistantMessages.length > 0 && userMessages.length === 0) {
    source = 'assistant';
} else if (userMessages.length > 0 && assistantMessages.length === 0) {
    source = 'user';
}
// 如果两者都存在 → source 保持 'user'（合理但未文档化）
```

如果 conversation 同时包含 user 和 assistant 消息（最常见场景），source 始终为 `'user'`。这可能是预期行为（优先归因于用户），但代码中无注释说明。

## 三、架构级发现

### A-1: ContextEngine 是真正的上帝对象

`ContextEngine` 类（656 行，45+ 个方法）直接依赖 15 个模块：

```
ContextEngine
├── SQLiteStorageAdapter (硬编码)
├── Extractor
├── Recaller
├── createCompleteFn (LLM)
├── createEmbedFn / createBatchEmbedFn
├── runFusion (fusion 管线)
├── reflectOnTurn / reflectOnSession
├── createWorkingMemory / updateWorkingMemory / buildWorkingMemoryContext
├── runReasoning
├── runMaintenance
├── HookRegistry (6 类钩子)
├── export / import
├── getStats / healthCheck
└── logger
```

任何模块变更都可能影响 ContextEngine。建议按领域拆分为 `ExtractionService`, `RecallService`, `MaintenanceService`, `HealthService` 等。

### A-2: 从环境变量读取 API Key 但配置支持不完整

`llm.ts` 的 `createCompleteFn` 检查 `process.env.ANTHROPIC_API_KEY`（第 51 行）作为 fallback。但其他提供商（OpenAI、DashScope）没有类似的 env fallback——它们完全依赖 config 对象。不一致的配置源增加了部署复杂度。
