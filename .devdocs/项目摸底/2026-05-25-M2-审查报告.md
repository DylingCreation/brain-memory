## 二、调用链分析

```
index.ts (barrel export)
  ├── re-export: src/engine/context, src/store/, src/recaller/, src/extractor/, ...
  ├── re-export: src/scope/isolation (v1 types)

              │     └── import: openclaw-plugin.ts (default + named exports)
```

**关键发现**：存在一条 5 层深的导入链 (`index → register → wrapper → plugin → core`)，每层都在做 re-export。这是典型的过度抽象。

## 三、问题清单

### 🔴 P0 — 阻塞级

#### P0-1: `openclaw-plugin.js` 是死文件 — 加载即崩溃

**What**: 
```javascript
// openclaw-plugin.js (第 3 行)
import { BrainMemoryHandler } from './src/plugin/handler.js';
```
`src/plugin/handler.js`（及 `handler.ts`）**不存在**。该文件若被任何运行时加载将立即抛出 `ERR_MODULE_NOT_FOUND`。

**Why 这是问题**: 
- 该文件存在于仓库根目录且注释声明为 "entry point for OpenClaw to load the brain-memory plugin"
- 任何人在阅读代码库时可能误将其作为入口点，导致困惑
- 它是项目演化过程中的残留物（v1.0 旧版入口），但未被清理

**Root Cause**: 项目从 v1.x 的 `BrainMemoryHandler` 架构迁移到 v2.x 的 `BrainMemoryPluginCore + ContextEngine` 架构后，旧的 JS 入口文件未被删除。`.npmignore` 虽然排除了该文件（不会发布到 npm），但本地仓库中仍保留。

**Recommendation**: 
1. 删除 `openclaw-plugin.js`
2. 或至少添加废弃注释并使其为空导出，防止被误用

### 🟠 P1 — 高危级

#### P1-1: 三套插件入口共存，架构冗余严重

**What**: 项目存在三套不同的插件入口机制：

| 入口文件 | 状态 | 引用者 | 版本号 |
|----------|------|--------|--------|
| `openclaw-plugin.js` | 🔴 死文件（handler.js 不存在） | 无（未被引用） | 1.0.0 |
| `openclaw-plugin.ts` | ⚠️ 桥接层（仅 11 行 re-export） | `openclaw-wrapper.ts` | 继承 core |
| `openclaw-register.ts` | ✅ 当前标准入口 | `index.ts`, `package.json` | 1.8.0 (硬编码) |

**Why 这是问题**: 
- 开发者需要理解 3 个文件才能搞清楚插件入口机制
- `openclaw-plugin.ts` 作为中间桥接层没有实质逻辑，纯粹增加导入深度
- 三个文件的版本号各不相同：`openclaw-plugin.js` 声明 1.0.0，`openclaw-register.ts` 硬编码 1.8.0，`package.json` 声明 2.0.0

**Import chain depth**: 5 层
```
index → register → wrapper → plugin → core
```

**建议**: 
1. 删除 `openclaw-plugin.js`（P0-1）
2. 合并 `openclaw-plugin.ts` 到 `openclaw-wrapper.ts`（消除一层 re-export）
3. 统一版本号为单一来源（从 `package.json` 读取或使用常量）

#### P1-2: `openclaw-register.ts` 硬编码版本号 `1.8.0` 与 `package.json` 的 `2.0.0` 不一致

**What**: 
```typescript
// openclaw-register.ts:33
version: '1.8.0',

// package.json:3  
"version": "2.0.0",
```

**Why 这是问题**: 插件注册到 OpenClaw 时上报的版本号是 `1.8.0`，但实际包版本是 `2.0.0`。这会导致：
- 插件管理界面显示过期版本
- 版本依赖检查（`pluginApi >= 2026.3.24-beta.2`）可能使用错误的版本号
- 社区/用户报告 bug 时使用错误的版本号

**Root Cause**: 版本号分散在多个文件中，缺少单一事实来源（single source of truth）。`openclaw-register.ts` 在 `v1.8.0 F-7` 重构时创建，版本号被硬编码后未随 package.json 同步更新。

**建议**: 
1. 从 `package.json` 动态读取版本号：`import pkg from '../package.json' assert { type: 'json' }`
2. 或至少添加 pre-commit hook 检查版本号一致性

#### P1-3: `index.ts` 顶层 `console.log` — 模块加载即产生副作用

**What**: 
```typescript
// index.ts 最后一行
console.log('brain-memory module loaded successfully.');
```

**Why 这是问题**: 
- 每次 `import` 该模块都会在 stdout 输出日志，污染调用方的输出
- 在测试环境中，这条日志会出现在测试输出中
- 在服务端无头运行时，无人关心 "loaded successfully" 日志
- 违反纯导入不应有副作用的原则

**建议**: 移除该 `console.log`，或改为 `logger.debug()`（仅在 `DEBUG=brain-memory` 时输出）。

#### P1-4: `openclaw-wrapper.ts` 中大量使用 `any` 类型，丢失类型安全

**What**: 721 行的 wrapper 文件中出现大量 `any`：

```typescript
let storedConfig: any = null;                              // 第 41 行
export function register(api: any) { ... }                 // 第 138 行
export async function message_received(...args: any[]) { } // 第 326 行
// normalizeHookArgs 的 rawEvent 返回 any                // 第 298 行
sessionMemoryCache: Map<string, any>                       // 第 62 行
```

**Why 这是问题**: 
- `storedConfig: any` — 配置对象应该有明确的 `BmConfig` 类型
- `api: any` — OpenClaw Plugin SDK 的 API 类型应定义接口
- `...args: any[]` — 所有 hook 函数参数类型未定义

**Root Cause**: `openclaw-wrapper.ts` 是在 OpenClaw Plugin SDK 的 npm 包不可直接安装的环境下开发的。开发者为快速集成而选择了 `any` 类型绕过。

**建议**: 定义最小化的 OpenClaw Plugin API 接口类型（参考 `src/plugin/core.ts` 中已有的 `Message`, `SessionEvent` 等接口），至少覆盖 hook 函数签名。

#### P1-5: `normalizeHookArgs` 中 agentId 提取逻辑脆弱

**What**: 
```typescript
agentId: ctx?.accountId 
  || (typeof event?.sessionKey === 'string' ? event.sessionKey.split(':')[1] : undefined) 
  || 'default-agent',
```

**Why 这是问题**: 
- `event.sessionKey.split(':')[1]` 依赖 sessionKey 包含 `:` 分隔符且第二段是 agentId 的约定
- 如果 sessionKey 格式变更（如不含 `:` 或 agentId 不在 index 1），agentId 会静默回退到 `'default-agent'`
- 没有对 sessionKey 格式的断言或警告

**建议**: 添加格式验证。如果 `split(':')` 结果长度 < 2，记录 warn 日志而非静默使用 `'default-agent'`。

### 🟡 P2 — 中等级

#### P2-1: `DEFAULT_CONFIG` 在两个位置被合并，存在冲突风险

**What**: `FULL_DEFAULT_CONFIG` 在 wrapper 中被合并两次：

1. `register()`（第 161 行）：`storedConfig = { ...FULL_DEFAULT_CONFIG, ...bmConfig }` — 合并 OpenClaw 的运行时 plugin 配置
2. `init()`（第 217 行）：`const mergedConfig = { ...FULL_DEFAULT_CONFIG, ...(config || {}) }` — 合并 init 接收的 config 参数

然后（第 221 行）：`const effectiveConfig = storedConfig || mergedConfig`

**Why 这是问题**: 
- `register()` 和 `init()` 可能从 OpenClaw 接收到**不同的** config 对象
- `init()` 优先使用 `storedConfig`（即 register 的版本），意味着 init 独有的配置字段**会被丢弃**
- 如果 init 被调用时 register 尚未执行（如热重载场景），配置合并逻辑不同

| 场景 | 使用的配置 | 可能的 config 丢失 |
|------|-----------|-------------------|
| register 先于 init | storedConfig (register 的 config) | init config 的独有字段丢失 |
| init 先于 register | mergedConfig (init 的 config) | register config 独有字段丢失 |
| ensurePluginInitialized | storedConfig only | init config 完全不参与 |

**建议**: 统一配置合并路径，在单个位置完成 merge 并冻结结果。

---

#### P2-2: 每次消息被处理两次 — `handleMessage` + `getMemoryContext`

**What**: `message_received` hook（第 326 行）：
```typescript
// 第一次处理：提取知识
const result = await pluginInstance!.handleMessage(message);
// 第二次处理：检索记忆上下文
const memoryContext = await pluginInstance!.getMemoryContext(message);
```

**Why 这是问题**: 
- `handleMessage` 触发 `engine.processTurn()` → 执行 LLM 提取（可能很昂贵）
- `getMemoryContext` 触发 `engine.recall()` + `assembleContext()` → 又一次数据查询
- 在 high-traffic 场景下，每条用户消息触发两次完整的处理管线
- 这两个调用在功能上部分重叠：`handleMessage` 提取新知识写入图谱，`getMemoryContext` 从图谱检索已有知识注入上下文

**辨析**: 这不是 bug，是设计意图（提取 vs 召回是两个独立操作），但缺少文档说明这种"双通路"设计的理由。一次调用同时完成提取和召回更高效。

**建议**: 
1. 在 ADR 中记录此设计决策
2. 考虑 `message_sending` 钩子触发召回（在 AI 回复前），`message_received` 仅负责提取

#### P2-3: `_startHealthCheck` 中存在 AbortController 竞态条件

**What**: `src/plugin/core.ts:334-357`：
```typescript
private _startHealthCheck(): void {
  this._healthCheckAbort = new AbortController();
  const { signal } = this._healthCheckAbort;

  setTimeout(async () => {
    if (signal.aborted) return;
    const fn = createCompleteFn(this.config.llm);
    if (!fn) return;
    const timeoutId = setTimeout(() => this._healthCheckAbort!.abort(), 3000);
    // ...
  }, 1000);
}
```

**Why 这是问题**: 
- 如果在 `setTimeout` 的 1000ms 延迟期间调用 `shutdown()` → `this._healthCheckAbort.abort()` — 第二次 timeout 回调中的 `this._healthCheckAbort!.abort()` 使用 `!` 非空断言，但此时 `_healthCheckAbort` 可能已被 shutdown 设为 `null`
- 虽然实际崩溃概率低（需要恰好在 1000ms-4000ms 窗口内 shutdown），但类型安全已受损

**建议**: 在回调中捕获 `_healthCheckAbort` 的本地引用。

#### P2-4: `src/plugin/hooks.ts` 使用 v1 `ScopeFilter` 未迁移到 v2

**What**: `hooks.ts:15` 导入 `ScopeFilter`（v1 三层）而非 `ScopeFilterV2`（v2 六层）：
```typescript
import type { ScopeFilter } from '../scope/isolation';
export interface BeforeRecallInput {
  query: string;
  scopeFilter?: ScopeFilter;  // ← v1 类型，与 M1 发现的双 Scope 体系问题同源
}
```

**Why 这是问题**: 开发者钩子系统暴露了已过时的 v1 Scope 类型。用户注册的自定义钩子只能操作三层 scope，无法利用 v2 六层细粒度隔离。

**建议**: 在 Scope v1→v2 全量迁移中同步更新 hooks 类型。

#### P2-5: `message_sent` 钩子有内容长度过滤，但 `message_received` 没有

**What**: 
```typescript
// message_sent (第 423-424 行): 
if (trimmed.length < 50) return;  // 过滤短消息

// message_received (第 326 行): 
// 无过滤，任何长度都处理
```

**Why 这是问题**: 
- 用户发送 "ok"、"yes"、"no" 等短消息也会触发完整的 LLM 提取管线
- `message_received` 中没有调用 `isNoise()`（即使 `noiseFilter.enabled` 默认为 true），噪声过滤在 `message_received` 层未生效
- 不一致的过滤策略：对外发（AI 回复）有保护，对内收（用户输入）无保护

**Root Cause**: `message_received` 依赖 `plugin.handleMessage()` → `engine.processTurn()`，而 processTurn 内部的噪声过滤逻辑可能未在此路径生效。

**建议**: 在 `message_received` 入口也添加与 `message_sent` 一致的最小长度检查，或确认 `engine.processTurn()` 确实有噪声过滤。

### 🟢 P3 — 低等级

#### P3-1: 全局可变状态过多

**What**: `openclaw-wrapper.ts` 模块作用域中存在 6 个可变全局变量：

```typescript
let pluginInstance: BrainMemoryPluginClass | null = null;   // 插件实例
let storedConfig: any = null;                                // 运行时配置
let initPromise: Promise<void> | null = null;                // 初始化 Promise
let initComplete = false;                                     // 初始化完成标志
const sessionMemoryCache = new Map<string, any>();           // 记忆缓存 (200条上限)
const sessionMessagesBuffer = new Map<string, ...>();        // 消息缓冲 (100条/会话上限)
let registered = false;                                       // 重复注册防护
```

**Why 这是问题**: 7 个模块级可变状态使测试变得困难（需要在每个测试间重置状态），也使代码推理变得复杂。虽然有 `#8 fix` 注释表明曾修复过竞态条件，但全局状态本身就是竞态的根本原因。

**建议**: 将全局状态封装为一个单例类（如 `PluginRuntime`），在 `shutdown()` 和测试间可通过实例化/销毁来隔离状态。

---

#### P3-2: `sessionMessagesBuffer` 无过期清理

**What**: `pushSessionMessage()` 限制每个会话最多 100 条消息，但从未清理已结束会话的 buffer。如果 `session_end` 钩子未触发（如进程崩溃），缓冲的消息永远留在内存中。

**Why 这是问题**: 长期运行可能累积数百个已结束会话的消息缓冲，占用内存。

**建议**: 添加 TTL 机制（如 1 小时未访问则自动清理），或使用 WeakRef 模式。

---

#### P3-3: `platform` 提取逻辑模糊

**What**: `normalizeHookArgs` 中：
```typescript
platform: ctx?.channelId || event?.channel || null,
```

**Why 这是问题**: 
- `channelId` 和 `channel` 是不同概念：`channel` 可能是 Discord/Slack 等平台名，`channelId` 是平台内的具体频道
- 将 `channelId` 作为 `platform` 赋值在语义上是错误的 —— Discord 的频道 `#general` 不等于 platform `discord`
- 同时返回了 `channel`（原始字段）和 `platform`（复用 channelId），导致两个字段实际内容相同

**建议**: 区分 `platform`（如 "discord", "telegram"）和 `channelId`（如 "C123456"），不要混用。

## 四、架构级发现

### A-1: 5 层导入链的过度抽象

```
index.ts
 └── openclaw-register.ts      (66 行, 纯 re-export)
      └── openclaw-wrapper.ts  (721 行, 核心集成逻辑)
           └── openclaw-plugin.ts (11 行, 纯 re-export)
                └── src/plugin/core.ts (397 行, 真正的实现)
```

- 5 层中有 2 层（register、plugin）只做 re-export，无实质逻辑
- 每个中间层都维护了自己的类型/接口副本（`Message` 在 core.ts、wrapper.ts 各定义一份）
- `openclaw-plugin.ts` 只有 11 行，可以内联到 wrapper.ts 中直接 import core

### A-2: 配置传递的 "漏斗" 效应

配置从 OpenClaw Gateway → wrapper → plugin → engine 的过程中，经过多次浅拷贝合并：
```
OpenClaw config schema → api.config → storedConfig → createBrainMemoryPlugin(config) 
  → ContextEngine(config) → this.config
```

每一层都可能丢失深层嵌套字段。例如 `BmConfig.decay.betaCore` 如果只在 FULL_DEFAULT_CONFIG 中定义，但在某次 `{...config}` 浅拷贝时可能因 `config.decay` 对象被替换而丢失。

### A-3: `index.ts` 暴露了过多的内部实现细节

`index.ts` re-export 了 40+ 个函数/类型，包括低级工具函数（`escapeXml`, `cosineSimilarityF32`, `tokenize`, `jaccardSimilarity`），这些不是用户通常需要的 API。作为公共 API 入口，应该只暴露高级别接口：

```
✅ ContextEngine, Recaller, Extractor, DEFAULT_CONFIG, initDb
❌ escapeXml, cosineSimilarityF32, tokenize, jaccardSimilarity
```
