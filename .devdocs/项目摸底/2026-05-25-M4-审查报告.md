### 🟠 P1 — 高危级

#### P1-1: `runMaintenance` 忽略 `computeGlobalPageRank` 返回的 topK，自行重算且丢失节点名称

**What**: `pipeline.ts:88-95`：
```typescript
const pr = computeGlobalPageRank(storage, cfg);  // pr.topK 包含正确的节点名称
pagerankResult = {
  scores: pr.scores,
  topK: Array.from(pr.scores.entries()).sort(...).slice(0, 20)
    .map(([id, score]) => ({ id, name: id, score })),  // ← name = id!
};
```

`computeGlobalPageRank()` 内部已经正确计算了 `topK`（包含节点名称），但 `runMaintenance` 丢弃了它：
```typescript
// pagerank.ts:131 (正确的实现)
const sorted = Array.from(rank.entries()).sort(...).slice(0, 20)
  .map(([id, score]) => ({ id, name: nameMap.get(id) || id, score }));
```

**Why 这是问题**: 
- `runMaintenance` 的返回值被 `ContextEngine` 用于 UI 展示和统计
- `name: id` 意味着 UI 上将显示 `n-1715000000-abc12` 而不是 `"Python Error Handling Pattern"`
- 信息完全丢失

**Root Cause**: 代码重复——`computeGlobalPageRank` 已经计算了 topK，但 `runMaintenance` 没有复用其结果。增量 PageRank 路径也有同样的问题。

**Recommendation**: 直接使用 `pr.topK`：
```typescript
pagerankResult = { scores: pr.scores, topK: pr.topK };
```

#### P1-2: `pagerank.ts` 模块级缓存导致多实例冲突

**What**: `pagerank.ts:18-19`：
```typescript
let _cached: GraphStructure | null = null;
const CACHE_TTL = 30_000;
```

**Why 这是问题**: 
- 如果创建两个 `ContextEngine` 实例（例如不同 dbPath），两个实例共享同一个图缓存
- 实例 A 加载了 A 的图结构到缓存，实例 B 的 `personalizedPageRank` 会读到 A 的图
- 缓存是模块级的，不区分数据库路径或实例

**Root Cause**: 缓存设计时假设单实例场景。v2.0 引入多 scope 后可能产生多实例。

**Recommendation**: 将缓存从模块级移到 `IStorageAdapter` 实例级别，或通过 `Map<dbPath, GraphStructure>` 按路径隔离。

#### P1-3: `pipeline.ts` 中 4 处 `storage as any` 强制类型转换

**What**: `pipeline.ts:107, 117, 118` 等多处：
```typescript
summarizeCommunities(storage as any, cr.communities, llm, embedFn)
```

`storage` 已经是 `IStorageAdapter` 类型，而 `summarizeCommunities` 的第一个参数就是 `IStorageAdapter`。`as any` 不仅不必要，还绕过了类型检查。

**Why 这是问题**: 如果将来 `summarizeCommunities` 接口变更（例如需要 `DatabaseSyncInstance` 而非 `IStorageAdapter`），类型检查不会发现调用不兼容，导致运行时崩溃。

**Recommendation**: 删除所有不必要的 `as any`。

### 🟡 P2 — 中等级

#### P2-1: `buildLshBuckets` 使用确定性位置选择而非随机投影

**What**: `dedup.ts:36-43`：
```typescript
function buildLshBuckets(vectors, numBits = 8): Map<string, number[]> {
  for (let i = 0; i < vectors.length; i++) {
    const v = vectors[i].embedding;
    let sig = '';
    for (let b = 0; b < numBits; b++) {
      const idx = ((b * 131 + 7) * 3) % v.length;  // ← 固定位置
      sig += v[idx] >= 0 ? '1' : '0';
    }
  }
}
```

**Why 这是问题**: 
- 标准 LSH 使用随机超平面投影（随机向量点积 → 正负 sign 决定 bit）
- 当前实现从 embedding 的 8 个固定位置采样，如果这些位置的值为 0（对稀疏 embedding 很常见），所有向量都得到相同签名 → 所有向量落入同一 bucket → LSH 退化为 O(n²) 全量比较
- `numBits` 固定为 8，仅生成 256 个可能的 bucket，对万级节点不够

**Root Cause**: 这是一个简化的 LSH 近似，为性能而牺牲了正确性保证。但对于 1024 维的 dense embedding（如 bge-m3），固定位置采样的退化概率较低。

**Recommendation**: 
1. 添加注释说明这是简化版 LSH（非标准随机投影）
2. 监控 bucket 大小分布，当最大 bucket 超过总数 50% 时 warn
3. 对于 `numBits` 不足的场景，动态调整到 16 或 32

#### P2-2: `runIncrementalPageRank` 的归一化逻辑存疑

**What**: `pagerank.ts:219-225`：
```typescript
const dirtySum = Array.from(newRank.entries())
  .filter(([id]) => dirtyNodes.has(id)).reduce((s, [, v]) => s + v, 0);
if (dirtySum > 0) {
  const targetSum = dirtyRatio;  // ← 用脏节点比例作为目标总分
  const scale = targetSum / dirtySum;
  for (const [id, val] of newRank) {
    if (dirtyNodes.has(id)) newRank.set(id, val * scale);
  }
}
```

**Why 这是问题**: 
- PageRank 的数学性质要求所有节点分数之和为 1
- 此处将脏节点分数缩放到 `dirtyRatio`（例如 0.05），意味着脏节点总分只有 0.05，而边界节点的分数未调整
- 最终写入存储的分数不符合 PageRank 的数学定义

**Recommendation**: 
1. 如果要保持增量结果与全局结果一致，应该做全局归一化（所有节点分数除以总和）
2. 或在 ADR 中记录当前近似策略及其误差范围

#### P2-3: `scoreDecay` 基于 `createdAt` 而非 `lastAccessedAt`

**What**: `decay/engine.ts:35`：
```typescript
const ageDays = (n - node.createdAt) / MS_PER_DAY;
```

**Why 这是问题**: 
- 一个创建于 30 天前但每天都访问的节点，与一个创建于 30 天前从未访问的节点有相同的 `ageDays`
- 衰减模型已经通过 `frequency` 维度考虑了访问频率，但 `recency` 维度应该反映"上次访问距今"而非"创建距今"
- 这导致频繁使用的记忆被过度衰减

**对比**: `applyTimeDecay` 也使用 `createdAt`。两者逻辑一致但语义可能不是预期。

**Recommendation**: 考虑使用 `lastAccessedAt` 或 `updatedAt` 替代 `createdAt`，或在 ADR 中说明选择 `createdAt` 的理由。

---

#### P2-4: `cosineSim` 中有无效的 `!== undefined` 检查

**What**: `dedup.ts:24`：
```typescript
if (a[i] !== undefined && b[i] !== undefined) {
```

**Why 这是问题**: `Float32Array` 的元素永远不会是 `undefined`——未初始化元素默认为 0.0。这个检查增加了不必要的分支，在热点循环中影响性能。

**Recommendation**: 删除此检查。

### 🟢 P3 — 低等级

#### P3-1: 社区检测使用 `Math.random()` 非确定性排序

**What**: `community.ts:47-49` 和 `community.ts:175-178`：
```typescript
const shuffled = [...nodeIds];
for (let i = shuffled.length - 1; i > 0; i--) {
  const j = Math.floor(Math.random() * (i + 1));
  [shuffled[i], shuffled[j]] = [shuffled[j], shuffled[i]];
}
```

**Why 这是问题**: 
- 每次运行社区检测产生不同的结果（LPA 对节点处理顺序敏感）
- 这使得测试难以验证——相同输入产生不同输出

**Note**: 这是 LPA 算法的已知特性，不是 bug。但项目需要意识到：社区 ID `c-1`, `c-2` 在不同运行间不持久。

**Recommendation**: 
1. 在函数注释中明确声明"结果非确定性"
2. 测试中使用固定 seed 或接受结果不唯一

---

#### P3-2: `NOISE_RE` 仅匹配行首（`^`），可能漏过滤

**What**: `noise/filter.ts:10`：
```typescript
const NOISE_RE = /^(ok|okay|k|yes|no|yep|nope|sure|好的|收到|嗯嗯|哈哈|呵呵|👍|👌|✅|❤️)/i;
```

**Why 这是问题**: 
- `"I think ok, let's proceed"` 不会被过滤
- `"好的，我明白了"` 会匹配 → OK
- 但 `"那好的，我去做"` 不会匹配 → 可能漏过滤

**Note**: 这是一个设计权衡——`^` 锚点的使用避免了误杀有意义的消息（如 "Okay, based on your feedback, I'll..." 不应被过滤）。当前行为是保守的，不是 bug。

---

#### P3-3: `DYNAMIC_PATTERNS` 可能误分类学习类内容

**What**: `temporal/classifier.ts:24-25`：
```typescript
/\b(changed|moved|renamed)\b/i,  // 动态
```

**Why 这是问题**: 在代码学习上下文中，"the API was renamed from X to Y" 是一个**静态事实**（API 的名字变了，这是一个持久的历史信息），不应被分类为动态内容从而加速衰减。

**Recommendation**: 将 `changed/moved/renamed` 从动态模式中移除，或在分类逻辑中增加对 "version"/"release" 等版本上下文的检查。

## 三、架构级发现

### A-1: 维护管线的"双入口"设计

```
runMaintenance() in maintenance.ts
    └── 委托到 → runMaintenance() in pipeline.ts
```

`maintenance.ts` 的 `runMaintenance` 直接委托给 `pipeline.ts` 的同名函数，没有任何附加值。这是一个从旧结构（三路径函数）迁移到新结构（可组合管线）时的过渡产物。现在 `maintenance.ts` 只剩 `shouldRunIncremental` 和委托函数，可以考虑合并。

### A-2: 衰减模型的级联分层设计

衰减系统有三层结构：

| 层 | 文件 | 职责 |
|---|------|------|
| **数学引擎** | `engine.ts` | Weibull 公式实现 |
| **配置预设** | `presets.ts` | 4 种预设 + 可视化 |
| **管线集成** | `pipeline.ts` | `runDecayArchiving` 阈值触发 |

`DECAY_THRESHOLD = 0.25` 硬编码在 `pipeline.ts` 中（而非 `DecayConfig`），导致它不可配置。如果预设 `aggressive` 比 `balanced` 更快衰减，用户期望 aggressive 的归档阈值也相应降低，但目前做不到。

### A-3: graphWalk 的增量维护链

增量维护链：`markDirty → getAffectedSubgraph → getDirtyNodes → incremental PR/Community`

这个链依赖 `getAffectedSubgraph` 返回正确的子图。但在 LanceDB 适配器中，`getAffectedSubgraph` 使用 BFS（非递归 CTE），且可能因缓存不一致而遗漏边。当 dirty ratio 恰好落在阈值边界时，可能触发路径选择错误。
