## 三、问题清单

### 🔴 P0 — 阻塞级

#### P0-1: `migrateToV2_ScopeUpgrade` 中 scope_id 生成算法与 `computeScopeId()` 不一致

**What**: 迁移脚本和 JS 端使用了两种不同的算法生成 scope_id：

| 位置 | 算法 | 示例（platform=qqbot） |
|------|------|------------------------|
| `migrate.ts:117` (SQL) | `lower(hex(CONCAT(...)))` | `7071626f742e2e2e`（ASCII 的 hex 编码） |
| `isolation.ts:123` (JS) | `sha256(...).hex.slice(0,16)` | `a1b2c3d4e5f6a7b8`（哈希值） |

**Why 这是问题**: 
- 两种算法产生**完全不同**的 scope_id 值
- 旧数据（通过迁移生成）的 scope_id 无法与新数据（通过 `computeScopeId()` 生成）匹配
- 这会破坏 scope 隔离机制：同一 scope 的旧节点和新节点有不同的 scope_id，导致 scope 过滤查询无法找到旧节点

**Root Cause**: SQLite 没有内置 sha256 函数，迁移脚本退而选择 `hex(COALESCE(...))` 作为"简易 hash"。但这与 JS 端的 sha256 不兼容。

**Impact**: 任何已运行过 v2 迁移的数据库，其所有现有节点的 scope_id 都是错误的。scope 过滤查询将静默排除这些节点。

**Recommendation**: 
1. 迁移脚本中调用 Node.js 的 `crypto.createHash('sha256')` 为每行生成正确的 scope_id（通过参数化 UPDATE 逐行处理）
2. 或在小数据量场景下，`initDb()` 后在 JS 层重新计算全部节点的 scope_id

#### P1-2: `_helpers.ts` 中 `toNode()` 的 `source` 字段缺少 `'manual'` 类型

**What**: 
```typescript
// _helpers.ts:15
source: (r.source as string) as 'user' | 'assistant',
```

但 `BmNode` 接口定义：
```typescript
// types.ts
source: 'user' | 'assistant' | 'manual';
```

**Why 这是问题**: 任何通过 `INSERT` 写入 `source='manual'` 的节点，通过 `toNode()` 读取时会被 TypeScript 类型系统视为 `'user' | 'assistant'`。虽然运行时不会崩溃（数据仍在），但：
- 类型窄化后 IDE 无法提示正确的 discriminant union
- 如果有代码检查 `node.source === 'manual'`，TypeScript 可能报告"此条件永远为 false"

**Impact**: `sqlite-adapter.ts:82` 中 `toNodeFromRaw` 函数有同样的问题：
```typescript
source: (r.source as string) as 'user' | 'assistant',
```

**Recommendation**: 修正为 `as 'user' | 'assistant' | 'manual'`。

#### P1-3: `upsertNode` 中 v1 scope 回退逻辑与 v2 语义冲突

**What**: 
```typescript
// nodes.ts:61
const scopeSession = c.scopeSession ?? sessionId;
const scopeChat = c.scopeChat ?? c.scopeSession ?? sessionId;
```

当调用方只传 v1 `scopeSession` 时，`scopeChat` 会正确回退到 `scopeSession`。但 `scopeSession` 本身也被默认设置为 `sessionId`。这意味着：

**Why 这是问题**: 如果调用方未提供任何 scope 信息，`scopeSession` 和 `scopeChat` 都会被设置为 `sessionId`（当前会话 ID）。而在 v2 六层 scope 设计中，`scopeChat` 是"会话/频道"的概念，`scopeSession` 已标记为 `@deprecated`。两者同时被设为 sessionId 意味着：
- 旧字段和数据迁移产生不一致 — 一个节点同时有 v1 和 v2 的默认值
- 调用方可能期望 `scopeSession` 为 null（因为已废弃），但实际上被隐式填充

**Recommendation**: 移除 `scopeSession` 的自动回退逻辑，仅在新字段（`scopeChat`）上应用默认值。

#### P2-2: `uid()` 使用 `Math.random()` 而非 `crypto.randomUUID()`

**What**: 
```typescript
// _helpers.ts:6
export function uid(prefix: string): string {
  return `${prefix}-${Date.now()}-${Math.random().toString(36).slice(2, 7)}`;
}
```

**Why 这是问题**: 
- `Math.random()` 不是密码学安全随机源，在同一毫秒内生成多个 ID 时碰撞概率更高
- Node.js 18+ 原生支持 `crypto.randomUUID()`，无需额外依赖
- 当前算法：`n-1715000000-abc12` — 毫秒精度 + 6 位 base36 (~2^31 种可能)，在高并发写入场景下存在碰撞风险

**建议**: 改用 `crypto.randomUUID()` 或至少增加随机位长度。

#### P2-3: `bm_nodes_fts` 外部内容 FTS5 的 UPDATE 触发器效率低

**What**: `db.ts` 中 `bm_nodes_au` 触发器：
```sql
CREATE TRIGGER IF NOT EXISTS bm_nodes_au AFTER UPDATE ON bm_nodes BEGIN
  INSERT INTO bm_nodes_fts(bm_nodes_fts, rowid, name, description, content)
  VALUES ('delete', old.rowid, old.name, old.description, old.content);
  INSERT INTO bm_nodes_fts(rowid, name, description, content)
  VALUES (new.rowid, new.name, new.description, new.content);
END;
```

**Why 这是问题**: 每次 UPDATE 执行两次 INSERT（先 delete 再 insert），即使 name/description/content 字段没有变更。对于仅更新 `pagerank` 或 `access_count` 的操作，FTS5 索引重建是无效的。

**建议**: 触发器中添加条件判断 `WHEN old.name != new.name OR old.description != new.description OR old.content != new.content`。

#### P2-4: `getEpisodicMessages` 中对每个 sessionId 执行一次子查询

**What**: 
```typescript
for (const sid of sessionIds) {
  const msgs = db.prepare(`
    SELECT role, content FROM bm_messages
    WHERE session_id=? ...
    ORDER BY ABS(turn_index - (
      SELECT turn_index FROM bm_messages WHERE session_id=? ...
      ORDER BY ABS(created_at - ?) LIMIT 1
    )) LIMIT 10
  `).all(sid, sid, nearTime) as SqlRow[];
```

**Why 这是问题**: 
- 对每个 sessionId 执行一个嵌套子查询（O(n²) 复杂度）
- 当 sessionIds 数量较多时（如 10+），产生大量数据库往返
- 子查询 `SELECT turn_index ... ORDER BY ABS(created_at - ?) LIMIT 1` 每次全表扫描 bm_messages

**建议**: 先批量获取每个 session 的 nearest turn_index，再统一查询消息。

#### P2-5: `vectorSearchWithScore` 全量加载再 JS 计算 cosine

**What**: `storage/search.ts:61-97`：
```typescript
// 1. 先加载所有 active nodes
const nodes = db.prepare(`SELECT * FROM bm_nodes WHERE status='active'...`).all();
// 2. 再加载它们的所有向量
const vectorRows = db.prepare(`SELECT ... FROM bm_vectors WHERE node_id IN (...) ...`).all();
// 3. JS 层逐个计算 cosine 相似度
for (const r of vectorRows) { /* O(n*dim) */ }
```

**Why 这是问题**: 当 active nodes 数量达到万级时，这种"全量加载+JS 计算"模式会：
- 将全部向量数据加载到内存（每个 1024 维 × 4 字节 × N 个节点）
- 在单线程中计算所有 cosine 距离

**Root Cause**: SQLite 没有原生向量搜索能力。这是使用 SQLite 作为向量存储的固有限制。`ISearchIndex`（LanceDB）的引入就是为了解决此问题。

**建议**: 在文档中明确 SQLite 向量搜索的规模限制（建议 < 10,000 节点），超出时建议使用 LanceDB 后端。

#### P3-2: LanceDB 适配器 `findNodeByName` 使用 O(n) 线性扫描

```typescript
findNodeByName(name: string) { 
  for (const n of this._nodeCache.values()) if (n.name === name) return n;
```

## 四、架构级发现

### A-1: 双存储后端的"功能一致性子集"问题

```
             IStorageAdapter (49 methods)
                    │
    ┌───────────────┴───────────────┐
    │ SQLiteAdapter ✅              │ LanceDBAdapter 🔶
    │ 49/49 方法完整实现            │ 29/49 有真实现
    │                               │ 11/49 是 stub
    │                               │ 9/49 是缓存操作（无持久化）
```

LanceDB 适配器是 `@lancedb/lancedb` 处于 `devDependencies` 的 POC。但它在 `IStorageAdapter` 接口层面承诺了"完整实现"。当一个模块通过 `IStorageAdapter` 调用 `getCommunity()` 时，它无法判断底层是返回真实数据还是 null。

**设计建议**: 引入 `Capability` 标志位，让上层代码能查询存储后端支持哪些功能：
```typescript
interface IStorageAdapter {
  capabilities: { community: boolean; messages: boolean; vector: boolean; ... };
}
```

### A-2: `StorageFilter` 作为双 Scope 体系的胶水层

`src/store/adapter.ts:59-70` 的 `StorageFilter` 同时包含 v1 和 v2 的 scope 字段：
```typescript
includeScopes?: MemoryScope[];        // v1 (deprecated)
excludeScopes?: MemoryScope[];
includeScopesV2?: MemoryScopeV2[];    // v2
excludeScopesV2?: MemoryScopeV2[];
```

这延续了 M1 发现的 v1/v2 双体系问题。存储层在 sqlite-adapter 中需要 `scopeFilterToStorageFilter()` 转换 v1，然后用 `extractScopeFilterV2()` 提取 v2，最终同时传递给底层函数。4 层胶水代码增加了维护成本。
