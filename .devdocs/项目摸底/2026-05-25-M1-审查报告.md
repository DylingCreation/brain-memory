### 🔴 P0 — 阻塞级（必须立即处理）

#### P0-1: `.env` 文件包含明文 GitHub Token

**What**: `/home/yangchen/.openclaw/codinghelper/brain-memory/.env` 文件第 14 行包含 `GITHUB_TOKEN=<REDACTED>`。

**Why**: 这是一个拥有仓库访问权限的 GitHub Personal Access Token，以明文形式存储在工作目录中。虽然 `.env` 已通过 `.gitignore` 排除出 Git 版本控制，但该令牌仍存在于文件系统中，任何能访问此工作目录的人/进程都可以读取。

**Root Cause 分析**:
- **表层原因**: 测试脚本/开发者手动配置了真实 Token 到 `.env` 文件
- **深层原因**: 项目缺少 `.env` 文件的敏感信息检测机制（pre-commit hook / CI 检查）。`.env.example` 模板中未警告用户不要将真实凭据写入 `.env` 文件

**Impact**: 
- 该 Token 已暴露在文件系统中
- 虽未纳入 Git（已通过 `.gitignore` 保护），但任何本地恶意进程均可读取
- 该 Token 可能对应 `DylingCreation` GitHub 账号的仓库写权限

**Recommendation**: 
1. 立即在 GitHub Settings → Developer settings → Personal access tokens 中吊销该 Token
2. 从 `.env` 中删除该行
3. 在 `.env.example` 中添加安全提示注释
4. 考虑启用 Git hooks 中的 `detect-secrets` 或 `gitleaks` 扫描

### 🟠 P1 — 高危级（必须在当前版本修复）

#### P1-1: 双 Scope 体系共存 — v1 → v2 迁移不完整

**What**: 代码库中存在两套并行的 Scope 类型体系：

| 体系 | 类型 | 层级 | 定义位置 |
|------|------|------|----------|
| v1 | `MemoryScope`, `ScopeFilter` | 3 层 (sessionId/agentId/workspaceId) | `src/scope/isolation.ts` |
| v2 | `MemoryScopeV2`, `ScopeFilterV2` | 6 层 (platform/workspace/agent/user/chat/thread) | `src/types.ts` |

**Why 这是问题**:
- 所有存储层函数接受 `(scopeFilter?: ScopeFilter, scopeFilterV2?: ScopeFilterV2)` 双参数，每次调用需同时传递 v1 和 v2 过滤器
- `recaller/cache.ts` 被迫使用 `type AnyFilter = ScopeFilter | ScopeFilterV2` 这种绕过类型检查的 workaround
- v1 工具函数（`scopesMatch`, `scopeKey`, `buildScopeFilterClause`）仍被 `recaller/cache.ts`, `plugin/hooks.ts`, `store/storage/nodes.ts` 等模块使用——这些模块自己也不知道应该用 v1 还是 v2
- `src/store/sqlite-adapter.ts:36` 注释 "Convert StorageFilter to v1 ScopeFilter or v2 clause+params" 表明这是一个已知的临时方案

**Root Cause**: v2.0 Scope 六层架构是作为增量功能引入的。开发团队选择了"兼容共存"策略（通过 `StorageFilter` 中间层传递双参数），而非一次性完成全量迁移。这导致新旧代码混合，API 语义模糊。

**Impact 范围**:
- 直接影响: `src/recaller/`, `src/store/`, `src/plugin/`, `src/retriever/` 四个核心模块
- 间接影响: 任何新增的 scope 过滤逻辑必须同时支持 v1 和 v2 参数
- 每增加新的 scope 层级，双体系维护成本线性增长

**建议**: 制定 Scope 体系全量迁移计划，分 3 个批次：
1. 统一所有存储层函数签名，移除 v1 `ScopeFilter` 参数
2. 移除 `src/scope/isolation.ts` 中的 v1 类型和函数
3. 清理 `StorageFilter` 中的 v1 字段

#### P2-1: 运行时代码污染类型定义文件

**What**: `src/types.ts` 中包含大量运行时代码：

| 运行时构造 | 行数 | 应归属模块 |
|-----------|------|-----------|
| `MEMORY_CATEGORIES` 常量 | 15 | types.ts（可接受，用于类型推导） |
| `EDGE_FROM_CONSTRAINT` / `EDGE_TO_CONSTRAINT` | 40 | `src/graph/` 或独立 constraint 模块 |
| `DEFAULT_SCOPE_FILTER_V2` 常量 | 5 | `src/scope/isolation.ts` |
| `adaptScopeV1toV2()` 函数 | 10 | `src/scope/isolation.ts` |
| `adaptScopeV2toV1()` 函数 | 10 | `src/scope/isolation.ts` |
| `DEFAULT_CONFIG` 对象 | 85 | `src/config.ts` 或独立配置模块 |

总计约 165 行运行时代码存在于 `src/types.ts`（占文件 28%）。

**Why 这是问题**: 
- 违反关注点分离原则：类型文件应只包含接口/类型/枚举
- 运行时代码会增加类型文件的导入负担（任何 `import type { ... } from '../types'` 的文件仍会触发运行时代码的 tree-shaking 判断）
- `adaptScopeV1toV2`/`adaptScopeV2toV1` 函数放在 types.ts 中，定位困难（开发者不会在 types 文件中寻找适配器函数）

**建议**: 
- 将 `DEFAULT_CONFIG` 移至 `src/config.ts`
- 将 scope 适配函数移至 `src/scope/isolation.ts`
- 将 Edge 约束表移至 `src/graph/` 或独立 `src/constraints.ts`

#### P2-3: `FusionConfig` 可选字段在 `DEFAULT_CONFIG` 中未初始化

**What**: `FusionConfig` 接口定义了 4 个可选字段并附有默认值注释：
```typescript
namePreFilterThreshold?: number;    // default 0.2
nameWeight?: number;                 // default 0.6
vectorWeight?: number;               // default 0.4
autoMergeThreshold?: number;         // default 0.9
```
但 `DEFAULT_CONFIG.fusion` 中没有设置这些值：
```typescript

  similarityThreshold: 0.75,
  minNodes: 20,
  minCommunities: 3,
  // 缺少: namePreFilterThreshold, nameWeight, vectorWeight, autoMergeThreshold

```

**Why 这是问题**: 依赖"如果为 undefined 则在代码中回退到硬编码默认值"的模式容易引入 bug——如果有 3 个不同文件各自硬编码了默认值，修改时容易遗漏。

**建议**: 在 `DEFAULT_CONFIG.fusion` 中显式设置所有字段的默认值。

#### P2-4: `GraphNodeType` 与 `MemoryCategory` 的映射关系不明确

**What**: 
- `GraphNodeType` 只有 3 种：`'TASK' | 'SKILL' | 'EVENT'`
- `MemoryCategory` 有 8 种：profile, preferences, entities, events, tasks, skills, cases, patterns

问题：`profile`, `preferences`, `entities`, `cases`, `patterns` 这 5 个分类应该映射到哪种 `GraphNodeType`？

**Why 这是问题**: 
- 在 `BmNode` 接口中，`type: GraphNodeType` 和 `category: MemoryCategory` 是独立字段，但没有类型级别约束来保证 `category='profile'` 的节点必须使用哪个 `type`
- 可能导致数据不一致：profile 节点可能被随机赋予 TASK/SKILL/EVENT 类型
- 边约束表 (`EDGE_FROM_CONSTRAINT`, `EDGE_TO_CONSTRAINT`) 只基于 `GraphNodeType`，这意味着 profile/entities 等分类节点无法参与大多数图谱关系

**Root Cause**: 设计可能是"只有 TASK/SKILL/EVENT 类型的节点可以建立图谱边，其他 5 类节点仅作为属性节点存在"。但缺少明确的架构决策记录（ADR）来文档化这个设计意图。

**建议**: 
1. 在 types.ts 中添加 `category` → `type` 的映射表（如 `const CATEGORY_TO_TYPE: Record<MemoryCategory, GraphNodeType>`）
2. 或者扩展 `GraphNodeType` 以包含全部 8 种分类
3. 编写 ADR 记录此设计决策

#### P2-5: `buildScopeFilterClause` 中存在潜在的数据暴露风险

**What**: 在 `src/scope/isolation.ts:89-97`，当 `allowCrossScope=true` 且共享模式下，如果没有 `includeScopes` 限制：
```typescript
if (filter.includeScopes.length > 0) { ... } // 跳过
// 然后直接追加 cross-scope 条件
conditions.push(`(category IN (${placeholders}))`); 
```

**Why 这是问题**: 如果调用方设置 `allowCrossScope: true` 但忘记设置 `includeScopes`，可能导致跨 scope 的 unintended data exposure——虽然受限于 `mixed` 模式的 category 限制，但仍可能暴露本不应可见的数据。

**建议**: 当 `includeScopes` 为空时，应至少要求一个最低限度的过滤条件（如 `scope_agent` 或 `scope_chat`），或在此场景添加 warn 日志。

## 三、架构级发现

### A-1: Scope 架构的 "中间态" 特性

v2 Scope 六层架构是一个合理的演化方向（从会话级 → 六层级细粒度隔离），但当前代码处于一种"尴尬的中间态"：

```
   调用方 (plugin/core.ts, engine/context.ts)
        │
        ├──→ MemoryScopeV2 (六层) ──→ 各模块签名
        │
        └──→ StorageFilter ──→ sqlite-adapter
                 │
                 ├──→ ScopeFilter (v1 三层) ──→ buildScopeFilterClause()
                 └──→ ScopeFilterV2 (v2 六层) ──→ buildScopeFilterClauseV2()
```

v1 和 v2 的 filter 被同时传递到 `allActiveNodes()`, `searchNodes()`, `vectorSearchWithScore()` 等存储函数。每次调用都需要构造两个过滤器对象，增加了不必要的内存开销和代码复杂度。

**设计建议**: 统一为 v2 Scope（六层），废弃 v1 三层体系。`BmNode` 中保留的 `scopeSession` 字段（已标记 `@deprecated`）可以保留用于数据兼容，但 API 层应只接受 `ScopeFilterV2`。

---

### A-2: 全局可变状态 — `DEFAULT_CONFIG` 的共享风险

`DEFAULT_CONFIG` 是一个 `const` 对象，但其嵌套属性（如 `decay.betaCore`）是可变的。如果有任何代码执行 `DEFAULT_CONFIG.decay.betaCore = 0.5`，会影响所有后续引用。虽然当前没有这样的代码，但缺乏防御措施。

**建议方案**:
1. 使用 `Object.freeze()` 深度冻结 `DEFAULT_CONFIG`
2. 或在 `ContextEngine` 构造函数中使用 `structuredClone(DEFAULT_CONFIG)` 创建独立副本

## 四、项目级发现

### P-1: 代码组织 — `src/types.ts` 职责过重

578 行的 types.ts 承担了双重职责：
1. 类型定义（接口、类型别名、枚举）
2. 运行时默认配置 + 适配器函数

遵循单一职责原则，建议拆分为：
- `src/types.ts` — 纯类型定义
- `src/config.ts` — 默认配置实例
- `src/scope/adapter.ts` — scope 适配函数

### P-2: `openclaw.plugin.json` 与 `BmConfig` 的双重维护

插件配置 schema（`openclaw.plugin.json`）是对 `BmConfig` 的手工 JSON Schema 拷贝。当 `BmConfig` 新增字段时，plugin.json 必须手动同步更新。当前两者的字段基本一致，但缺少自动化同步机制。

### P-3: 文档备注 — `DEFAULT_CONFIG` 中版本标记风格不统一

部分配置项有版本标记（如 `// v1.0.0 B-3: default on`），部分没有（如 `compactTurnCount: 6` 无标记）。这给追溯配置变更历史带来困难。
