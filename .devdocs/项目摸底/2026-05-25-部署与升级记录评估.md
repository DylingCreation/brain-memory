#### ⭐ 最关键的单项发现：Ollama `think` 参数错误

```typescript
// ❌ 当前 brain-memory 代码 — DashScope 格式，Ollama 上静默失效
body.thinking = { type: 'disabled' };

// ✅ Ollama 原生 /api/chat 的正确做法
body.think = false;  // 顶层 bool，非嵌套对象
```

**影响量化**:

| 模块 | think=on 耗时 | think=off 耗时 | 加速比 | thinking 浪费 |
|------|-------------|---------------|--------|--------------|
| Extraction | 24.3s | 6.8s | 3.6× | 83% |
| Reflection | 58.5s | 4.2s | 14.0× | 93% |
| Fusion | 51.9s | 1.5s | 35.0× | 98% |
| Reasoning | 18.5s | 4.8s | 3.9× | 75% |
| **合计** | **153.2s** | **17.2s** | **8.9×** | **91%** |

> 之前反复出现但未被定位根源的"content 为空" bug，至此得到完整解释。

**每一层都有独立的验证闭环**，不在下一层依赖尚未稳定时推进。这完全符合《工程协作手册》的 "批次推进 + 质量门控" 原则。

### 🏆 亮点三：向后兼容设计堪称教科书

```typescript
// 三步走兼容策略

// 1. 类型层: 新旧 scope 类型共存
type MemoryScope    = { session, agent, workspace };      // v1
type MemoryScopeV2  = { platform, workspace, agent, user, chat, thread }; // v2

// 2. 兼容适配器: 自动转换
adaptScopeV1toV2(oldScope): MemoryScopeV2;
adaptScopeV2toV1(newScope): MemoryScope;

// 3. 数据库: fallback 过渡
// scopeChat 先读新字段 scope_chat，如果为 NULL 则读旧字段 scope_session
// 迁移前/迁移后的数据都能正确映射
```

这是迁移兼容设计的标准范式: **不破坏旧接口、不要求立即迁移、新旧并行、渐进切换**。
