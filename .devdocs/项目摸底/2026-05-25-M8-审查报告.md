### 🟠 P1 — 高危级

#### P1-1: UI 服务器使用 `readFileSync` 同步读取，阻塞事件循环

**What**: `server.ts:71-82`：
```typescript
const content = readFileSync(fullPath, 'utf-8');  // 同步 I/O
```

**Why 这是问题**: 
- 在 HTTP 请求处理器中调用同步文件 I/O 会阻塞 Node.js 事件循环
- 多个并发静态文件请求会排队等待，web UI 在高负载下响应变慢
- 这是 Node.js 性能指南中明确禁止的反模式

**Recommendation**: 使用 `fs.promises.readFile()` 或在启动时预加载静态文件到内存。

---

#### P1-2: `extractJson` 贪婪 `<think>` 正则可能吞噬全部内容

**What**: `json.ts:18`：
```typescript
s = s.replace(/<think>[\s\S]*/gi, '');  // 贪婪匹配！
```

**Why 这是问题**: 
- 如果 LLM 输出包含未闭合的 `<think>` 标签（如 `"<think>some text"` 缺少 `</think>`），贪婪匹配 `[\s\S]*` 会从 `<think>` 一直匹配到字符串末尾
- 结果是整个响应被清空 → `extractJson` 返回空字符串 → 所有 JSON 解析失败
- 第一行 `s.replace(/<think>[\s\S]*?<\/think>/gi, '')` 使用惰性匹配（`*?`）处理了闭合标签的情况——但第二行处理未闭合标签时用了贪婪匹配

**Root Cause**: 第二行意图是处理"有 `<think>` 但无 `</think>` 的遗留标签"，但 `[\s\S]*` 吃掉了标签后的所有内容。

**Recommendation**: 
```typescript
s = s.replace(/<think>[\s\S]*?$/gi, '');  // 惰性匹配到行尾，或
s = s.replace(/<think>[\s\S]*?(?=\{|$)/gi, '');  // 匹配到下一个 JSON 起始
```

#### P1-3: `config.ts` 使用 `require('fs')` 内联导入

**What**: `config.ts:88`：
```typescript
lastModified: existsSync(configPath)
  ? new Date((require('fs').statSync(configPath)).mtimeMs).toISOString()
```

**Why 这是问题**: 
- 文件顶部已通过 `import { statSync, ... } from 'fs'` 导入了 `existsSync`，但**没有导入 `statSync`**
- `require('fs')` 是一种 CJS/ESM 混合调用，在 ESM 项目中可能抛出 `require is not defined`
- `statSync` 可以通过顶部的 ESM import 添加

**Recommendation**: 在顶部添加 `statSync` 到 import 列表，移除 `require('fs')`。

### 🟡 P2 — 中等级

#### P2-1: `cosineSimilarity` 和 `cosineSimilarityF32` 的 `!== undefined` 冗余检查（M4/M5 同源）

**What**: `similarity.ts:13-14, 31-32`：与 M4 dedup.ts 和 M5 fusion/analyzer.ts 相同的模式：
```typescript
if (a[i] !== undefined && b[i] !== undefined) { ... }
```

`number[]` 和 `Float32Array` 的元素永远不会是 `undefined`。

**Recommendation**: 删除检查。这是代码库中该模式出现的第 4 处（前 3 处在 M4、M5），表明存在复制粘贴传播。

---

#### P2-2: `graph.ts` 控制器全量加载所有节点到内存

**What**: `graph.ts:18-21`：
```typescript
const allNodes = storage.findAllActive();  // ← 全量加载
const allEdges = storage.findAllEdges();
const sorted = allNodes.sort(...);
const nodes = sorted.slice(0, maxNodes);   // 然后用 JS 过滤
```

**Why 这是问题**: 当活跃节点数达到 10,000+ 时，每次 `/api/graph` 请求都会：
1. 将所有节点加载到内存（`findAllActive()` → 完整 `BmNode[]`）
2. 将所有边加载到内存
3. 在 JS 中排序 + 过滤

**Recommendation**: 在 IStorageAdapter 中添加 `findTopNodesByPageRank(limit)` 方法，或在 SQL 层完成排序/分页。

---

#### P2-3: `nodes.ts` 控制器中 `source: 'manual' as any` 绕过类型系统

**What**: `nodes.ts:109`：
```typescript
const result = storage.upsertNode({
  source: 'manual' as any,  // ← 强制类型转换
```

**Why 这是问题**: 
- `NodeUpsertInput.source` 定义为 `'user' | 'assistant' | 'manual'`（types.ts:229），按理应该是合法值
- 但 M3 P1-2 发现的 `_helpers.ts` 中 `toNode()` 的 `source` 类型断言为 `'user' | 'assistant'`（缺少 `'manual'`），导致 TypeScript 推断出不一致的类型
- 此处 `as any` 是为了绕过类型系统的不一致性

**Root Cause**: 根因在 M3 P1-2（`_helpers.ts` 缺少 `'manual'`），此处只是症状。

**Recommendation**: 修复 M3 P1-2 后移除此处的 `as any`。

### 🟢 P3 — 低等级

#### P3-1: `balanceBraces` 状态机复杂度高（70+ 行）

**What**: `json.ts:61-106`：`balanceBraces` 实现了一个完整的手工 JSON 括号平衡器，包含字符串/转义跟踪、括号栈、递归重入。

**Why 这是问题**: 
- 手工状态机容易出边界 bug（如 `\"` 转义引号、字符串内的 `{` `}`）
- 已有 3 个外部测试用例覆盖此函数（`json-enhanced.test.ts`, `json-extended.test.ts`），表明历史上出过 bug
- 复杂性由 LLM JSON 输出不稳定导致——如果能提升 prompt 质量减少 JSON 格式错误，此函数可大幅简化

**Recommendation**: 添加注释说明 `balanceBraces` 是最后防线，正常情况不会触发。定期审查是否可以移除。

---

#### P3-2: `logger.isEnabled` 每次调用重新读取环境变量

**What**: `logger.ts:67-69`：
```typescript
isEnabled(level: LogLevel): boolean {
  return LEVEL_PRIORITY[level] <= LEVEL_PRIORITY[getMinLevel()];
}
```

每次调用 `getMinLevel()` 都读取 `process.env.BM_LOG_LEVEL`。在频繁日志输出时产生不必要的 `process.env` 访问和字符串操作。

**Recommendation**: 在模块加载时缓存 `getMinLevel()` 的结果。

---

#### P3-3: `smallJsonRepair` 将非 JSON 文本强制包装为结论 JSON

**What**: `json.ts:159-162`：
```typescript
if (!raw.includes('{')) {
  const wrapped = `{"conclusions": [{"text": ${JSON.stringify(raw.slice(0, 200))}, "type": "implicit", "confidence": 0.5}]}`;
}
```

**Why 这是问题**: 如果 LLM 返回纯文本而非 JSON（如模型输出错误消息），此函数会将其包装为推理结论并以 `confidence: 0.5` 返回。这导致垃圾输入被当作有效结论。

**Recommendation**: 仅在 LLM 输出看起来"接近 JSON"时（如包含关键词 `conclusions` 或 `nodes`）才尝试包装。

## 三、架构级发现

### A-1: UI 控制器的 `findAllActive → JS filter` 模式

三个控制器（`stats.ts`, `graph.ts`, `nodes.ts`）都使用相同的模式：
```
findAllActive() → JS .filter() → JS .sort() → JS .slice()
```

这种模式将所有数据和计算压力转移到 Node.js JS 层，绕过了 SQLite 的索引排序和分页能力。对于小规模（< 1,000 节点）没问题，但大规模下（10,000+）会成为瓶颈。

### A-2: 工具函数质量整体较高

`json.ts`（容错解析）、`truncate.ts`（智能截断）、`tokens.ts`（中英文感知）、`logger.ts`（分级日志）实现了良好的工程实践。`balanceBraces` 复杂度是 LLM 输出不稳定性的合理代价。

### A-3: `cosineSimilarity` 实现的重复传播

```
similarity.ts (canonical)
  ↓ 被 reranker.ts 使用
  ↓ 但 fusion/analyzer.ts 有内联副本 (未 import)
  ↓ dedup.ts 有内联副本 (未 import)
  ↓ vector-recall.ts 没有使用余弦相似度

所有副本的 `!== undefined` 检查都被复制了
```

建议进行专项清理：全局搜索内联的 cosine similarity 实现，统一使用 `src/utils/similarity.ts`。
