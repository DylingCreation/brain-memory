## 三、问题清单

### 🔴 P0 — 阻塞级

#### P0-1: Lint 质量回归 — 从 0 error 恶化到 34 errors

**What**: 
- 历史基线（2026-05-20）：**0 errors, 122 warnings**
- 当前状态：**34 errors**（21 `no-explicit-any` + 12 `no-unused-vars` + 1 `no-empty`）

**Why 这是问题**: 
- `@typescript-eslint/no-explicit-any` 错误表示代码中使用了 `any` 类型（已在 M2 P1-4 发现），lint 层面未被修复
- `@typescript-eslint/no-unused-vars` 错误表示存在 12 处未使用的变量——可能是重构遗留或死代码
- 这是 v2.0.0 发布后引入的质量回归

**Root Cause**: v2.0.0 scope 升级和其他新增功能引入了大量 `any` 类型绕过（如 M1 的 `adaptScopeV2toV1`、M2 的 wrapper 层、M7 的 config 合并）。开发过程中未同步修复 lint。

**Recommendation**: 
1. 在下一个迭代中分配 lint 修复任务（预计 2-4 小时工作量）
2. 将 lint 错误数设为 CI 质量门（当前设为 0 error）
3. 逐个修复 `no-unused-vars`（最简单）→ `no-empty` → `no-explicit-any`（需重构）

### 🟠 P1 — 高危级

#### P1-1: 37 个源模块无专用测试 — 核心引擎和存储适配器无覆盖

覆盖缺口分布：
```
🔴 关键模块（7 个，2,756 行）: ContextEngine, SQLiteAdapter, IStorageAdapter, 
   BrainMemoryPluginCore, Recaller, FusionAnalyzer, CommunityDetection
🟠 重要模块（5 个，984 行）: ScopeIsolation, Pipeline, JSON utils, 
   WorkingMemory, AssembleContext
🟡 次要模块（25 个，~1,000 行）: storage/*, ui/*, utils/*
```

**Why 这是问题**: 
- `ContextEngine`（656 行）是系统中枢，无任何单元测试意味着任何回归都只能通过集成测试或手动测试发现
- `SQLiteAdapter`（412 行）有 49 个方法实现，无测试意味着 adapter 的纯委托逻辑中隐藏 bug
- `Recaller`（398 行）没有基础测试（只有 extended 测试），recall 缓存和种子采集逻辑未单独验证

**Recommendation**: 
优先为前 7 个关键模块编写单元测试。这是阻断 CI 质量门的最高优事项。

---

#### P1-2: 36 个测试文件（50%）无 `afterEach` 资源清理

**What**: 72 个测试文件中有 36 个缺少 `afterEach` 或 `beforeEach` 清理。

**Why 这是问题**: 
- 数据库测试文件（如 `store.test.ts` 使用 `:memory:` 无清理问题，但使用 `createTestStorage()` 的会创建临时文件）
- 无清理的文件可能导致：
  - 临时数据库文件残留（占用磁盘）
  - 测试间状态泄漏（如果使用共享数据库路径）
  - 内存泄漏（缓存/全局状态）

**对比**: 工程协作手册第 9.2 条规定："每个测试负责自身资源；beforeEach 清理 + afterEach 清理"

**Recommendation**: 为所有使用临时文件的测试添加 `afterEach` 清理，使用 `cleanupTestDb()` helper。

### 🟡 P2 — 中等级

#### P2-1: 92 处非确定性模式（`Math.random`/`Date.now`/`setTimeout`）

**What**: 测试代码中出现 92 处非确定性函数调用。

**Why 这是问题**: 
- `Math.random()` 在社区检测测试（LPA 算法）中使用——结果在不同运行间不同
- `Date.now()` 在时间敏感测试中使用——在 CI 慢机器上可能失败
- 这些是 flaky test 的潜在来源

**验证**: 当前 20 个跳过测试中可能部分是由于非确定性导致的。

**Recommendation**: 对 `Math.random()` 使用 vitest 的 `vi.spyOn(Math, 'random')` mock；对 `Date.now()` 使用 `vi.useFakeTimers()`。

---

#### P2-2: 测试文件扁平化——未按手册建议的组织结构

**What**: 所有 72 个测试文件平放在 `test/` 目录下。

**Why 这是问题**: 
- 工程协作手册第 9.1 条建议使用 `test/unit/`, `test/integration/`, `test/performance/`, `test/e2e/` 子目录
- `package.json` 的 test 脚本也预期此结构（`test:unit`, `test:integration` 等）
- M1 发现这些脚本指向不存在的目录（M1 P1-3）

**Recommendation**: 在修复 M1 P1-3 时同步重组测试文件。

### 🟢 P3 — 低等级

#### P3-1: `test-engine.ts` 是孤立烟雾测试，未纳入 vitest 套件

**What**: 根目录的 `test-engine.ts`（57 行）是一个独立的 smoke test 脚本，使用 `import './src/engine/context.ts'` 直接导入。

**Why 这是问题**: 
- 它不是 vitest 测试套件的一部分——不会在 `npm test` 中运行
- 它包含硬编码的 `process.env.OPENAI_API_KEY` ——不适用于 CI
- 它创建 `./test-brain-memory.db` 文件而无清理

**Recommendation**: 要么整合到 vitest 套件，要么将其转为 CI 专用的 smoke test（通过环境变量门控）。
