### 🟠 P1 — 高危级

#### P1-1: `evaluateSessionValue` 逻辑顺序错误 — `archive` 分支永远无法到达

**What**: `session/compressor.ts:60-66`：
```typescript
if (valueScore < 0.2 && messages > 20) {
  compressRecommendation = 'compress';   // ← 先匹配
} else if (valueScore < 0.1 && messages > 50) {
  compressRecommendation = 'archive';    // ← 永远无法到达！
}
```

**Why 这是问题**: 
- 如果 `valueScore=0.05` 且 `messages=60`，先被第一个条件 `valueScore < 0.2 && messages > 20` 捕获，返回 `'compress'`
- 第二个条件 `valueScore < 0.1` 是第一个的**子集**（更严格），但被放在 `else if` 中
- `'archive'` 推荐在任何输入下都不会产生

**Root Cause**: 条件顺序错误。`valueScore < 0.1`（更严格的阈值）应该排在 `valueScore < 0.2` 之前。

**Recommendation**: 
```typescript
if (valueScore < 0.1 && messages > 50) {
  compressRecommendation = 'archive';
} else if (valueScore < 0.2 && messages > 20) {
  compressRecommendation = 'compress';
}
```

#### P1-2: `reflection/store.ts` 使用 `DatabaseSyncInstance` 直接操作数据库，绕过 `IStorageAdapter`

**What**: `reflection/store.ts` 直接导入 `DatabaseSyncInstance`：
```typescript
import { type DatabaseSyncInstance } from '@photostructure/sqlite';
// ...
export function storeReflectionInsights(db: DatabaseSyncInstance, ...) {
  db.prepare('UPDATE bm_nodes SET importance=...').run(...);
  db.prepare('INSERT INTO bm_nodes ...').run(...);
}
export function applyTurnBoosts(db: DatabaseSyncInstance, ...) { ... }
```

同样，`session/compressor.ts` 也直接使用 `DatabaseSyncInstance`。

**Why 这是问题**: 
- 这些函数完全绕过了 `IStorageAdapter` 抽象层
- 如果使用 LanceDB 后端，这些函数在运行时会失败（`IStorageAdapter` 没有 `prepare/run` 方法）
- 这是 M3 发现的 LanceDB 社区/消息 stub 问题的延伸——这次是反思和压缩系统

**Root Cause**: 这两个模块的开发早于 `IStorageAdapter` 抽象层的引入（v1.1.0 F-1），迁移时它们被遗漏了。

**Recommendation**: 将这两个文件改为接受 `IStorageAdapter` 参数，通过 adapter 的 `findNodeByName`、`upsertNode` 等方法操作数据。

---

#### P1-3: `parseFinalize` 中边类型过滤在 `correctEdgeType` 修正之前执行

**What**: `extract.ts:parseFinalize`：
```typescript
const newEdges = (p.newEdges ?? [])
  .filter((e) => e.from && e.to && e.type && VALID_EDGE_TYPES.has(e.type as EdgeType))
  // ↑ 如果 LLM 提供的类型不在 VALID_EDGE_TYPES 中，边在此被丢弃
  .map((e) => correctEdgeType(e as {...}, nameToType))
  // ↑ correctEdgeType 本来可以修正非法类型，但边已经被过滤掉了
  .filter((e) => e !== null);
```

**Why 这是问题**: 
- `correctEdgeType` 函数的作用就是"先检查类型是否合法，不合法则根据 from/to 节点类型修正"
- 但 `.filter` 在 `.map(correctEdgeType)` 之前执行，导致不合法类型的边被**提前丢弃**
- LLM 生成 `SOLVED_BY → TASK→TASK`（非法组合），`correctEdgeType` 可以修正为 `USED_SKILL`，但在 Finalize 路径中，这条边在修正之前就被过滤掉了

**对比**: `parseExtract`（第 172-182 行）正确地先 map 后 filter。`parseFinalize` 的 filter/map 顺序刚好反了。

**Recommendation**: 将 `.filter` 移到 `.map(correctEdgeType)` 之后，或移除提前过滤（因为 `correctEdgeType` 返回 `null` 时会被 `.filter(e !== null)` 处理）。

### 🟡 P2 — 中等级

#### P2-1: `findFusionCandidates` O(n²) 全量比较，无法扩展到万级节点

**What**: `fusion/analyzer.ts:78-108`：
```typescript
for (let i = 0; i < nodes.length; i++) {
  for (let j = i + 1; j < nodes.length; j++) {
    // 每对节点都做 name+vector 比较
    const nameScore = computeNameSimilarity(a.name, b.name);
```

**Why 这是问题**: 
- 10,000 个活跃节点 = ~50M 次比较
- `computeNameSimilarity` 对每对节点进行 `tokenize()` + `jaccardSimilarity()` 计算
- M4 的 `dedup.ts` 使用了 LSH 分桶来降低复杂度，但 `findFusionCandidates` 没有

**对比**: `dedup.ts` 的 `buildLshBuckets` 将 O(n²) 降为 O(n × bucket_size)，但 fusion 模块没有类似优化。

**Recommendation**: 复用 LSH 分桶逻辑，或只对同 community 内的节点对进行比较（利用 `findFusionCandidates` 已有 `communityCount` 阈值检查）。

---

#### P2-2: `heuristic.ts` 中 skills 正则规则与代码块/命令提取存在冗余

**What**: `heuristic.ts` 有三层提取：
1. `extractCodeBlocks()` — 提取 \`\`\`...\`\`\` → SKILL 节点
2. `extractCommands()` — 提取 npm/docker/git... 命令 → SKILL 节点
3. `HEURISTIC_RULES[5]` — 正则 `/\`\`\`|npm\s+|docker\s+|.../` → SKILL 节点

第 3 步的正则**包含了**第 1、2 步的模式。但 `seenNames` 去重机制阻止了重复插入。

**Why 这是问题**: 
- 虽然去重生效，但正则匹配仍被执行（浪费 CPU）
- 第 3 步的捕获使用 `m[0]` 而非 `m[1]` → 提取的 name 质量不如第 1/2 步

**Recommendation**: 从 `HEURISTIC_RULES` 的 skills 规则中移除代码块和命令相关模式，让专用的 `extractCodeBlocks`/`extractCommands` 处理。

#### P2-3: `cosineSimilarityF32` 中无效的 `!== undefined` 检查（与 M4 同源）

**What**: `fusion/analyzer.ts:30`：
```typescript
if (a[i] !== undefined && b[i] !== undefined) {
```

`Float32Array` 元素永远不会是 `undefined`。这是与 M4 `dedup.ts` 相同的冗余检查，在融合管线中额外增加了 ~50M 次不必要的分支判断（配合 P2-1 的 O(n²) 问题）。

**Recommendation**: 删除检查，直接 `dot += a[i] * b[i]`。

### 🟢 P3 — 低等级

#### P3-1: `UNSAFE_PATTERNS` 中 `[\s\S]{0,40}` 存在正则回溯风险

**What**: `reflection/extractor.ts:19`：
```typescript
/(?:reveal|print|dump|show|output|display)[\s\S]{0,40}(?:system\s+prompt|...)/i,
```

**Why 这是问题**: `[\s\S]{0,40}` 配合后面的 `(?:system\s+prompt|...)` 会产生惰性匹配回溯。如果输入中没有 "system prompt" 关键词，每个 "reveal"/"print" 都会尝试 0-40 个字符的窗口。这在大量短消息的处理中影响微乎其微，但代码质量上可优化。

**Recommendation**: 使用 `.split()` 预处理或限制回溯范围。

---

#### P3-2: `storeReflectionInsights` 查找关联节点时使用 O(n) 线性扫描

**What**: `reflection/store.ts:110-128` 的 `findRelatedNode` 函数：
```typescript
for (const node of nodes) {
  const nodeTokens = tokenize(`${node.name} ${node.content}`);
  const overlap = jaccardSimilarity(insightTokens, nodeTokens);
```

对每个 insight 遍历所有活跃节点计算 Jaccard 相似度。如果有 100 个 insight 和 10,000 个节点，这是 1M 次 tokenize+jaccard 调用。

**Recommendation**: 使用索引（如 category 预过滤）缩小搜索范围。

---

#### P3-3: `parseExtract` 中的类型转换模式混乱

**What**: `extract.ts:173-176`：
```typescript
if (!(n.description as string)) n.description = '' as string;
n.name = normalizeName(n.name as string) as unknown as string;
```

`as unknown as string` 双转换模式出现在多处，混淆了运行时赋值和类型断言的区别。

**Recommendation**: 使用显式类型断言或在 filter 阶段提供类型守卫。

## 三、架构级发现

### A-1: 三层提取架构的正确实现

```
消息进入
  │
  ├── Tier 1: heuristicExtract() ─── 规则匹配（<10ms，无需 LLM）
  │     └── extractCodeBlocks + extractCommands + HEURISTIC_RULES
  │
  ├── Tier 2: LLM extract() ─── LLM 提取（需要 API 配置）
  │     └── mergeResults(heuristic, llm)
  │
  └── Tier 3: parseExtractTolerant() ─── 容错回退（JSON 修复）
```

三层架构设计合理。Tier 1 始终运行作为底线，Tier 2 在 LLM 可用时补充精度，Tier 3 在 JSON 解析失败时兜底。`heuristicConfidence()` 函数可输出信心评分供监控使用。

### A-2: reflection/store.ts 的抽象泄漏

回顾整体存储抽象链：

```
ContextEngine → IStorageAdapter → SQLiteAdapter / LanceDBAdapter
                                      ↑ 正常工作
但是：
reflection/store.ts → DatabaseSyncInstance  ← 直接 SQLite（绕过 adapter）
session/compressor.ts → DatabaseSyncInstance ← 直接 SQLite（绕过 adapter）
```

这是存储层抽象的一个泄漏点。当前它被掩盖了，因为 `ContextEngine` 只创建 `SQLiteStorageAdapter`，但如果将来切换到 `LanceDBStorageAdapter`，反思和压缩功能将崩溃。

### A-3: fusion 模块的命名歧义

`fusion/analyzer.ts` 中的 `computeNameSimilarity` 函数使用 `tokenize` + `jaccardSimilarity`（Jaccard 系数）。但 `dedup.ts` 使用 `cosineSim`（余弦相似度）。两种去重管线（dedup 和 fusion）使用不同的相似度算法：

| 模块 | 名称相似度 | 内容相似度 |
|------|-----------|-----------|
| `dedup.ts` | 无 | 向量余弦 |
| `fusion/analyzer.ts` | Jaccard (token) | 向量余弦 |

dedup 不看名称只看向量，fusion 看名称+向量。这两个模块服务于不同的去重目的（语义去重 vs 知识融合），但缺乏文档说明它们的区别和使用场景。
