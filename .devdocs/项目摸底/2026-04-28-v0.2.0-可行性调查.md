## 一、调查范围

## 二、逐项验证结果

### F-1: 数据库迁移系统

| 维度 | 内容 |
|------|------|
| **现状** | `db.ts` 使用单一 `SCHEMA` + `CREATE TABLE IF NOT EXISTS`，无版本号、无迁移记录表 |
| **可行性** | ✅ 可行 |
| **方案** | 新增 `bm_meta` 表存储 schema_version，`initDb()` 后按版本递进执行 ALTER TABLE |
| **风险** | 低 — v0.2.0 仅涉及新增表/列，SQLite ALTER TABLE 完全覆盖 |
| **工作量** | 1 天 |

### F-2: 健康检查 API

| 维度 | 内容 |
|------|------|
| **现状** | 无健康检查方法，初始化失败时 throw 或 console.error |
| **可行性** | ✅ 可行 |
| **方案** | 新增 `healthCheck()` 方法，返回结构化 HealthStatus 对象 |
| **风险** | 极低 — 纯读取操作 |
| **工作量** | 0.5 天 |

### F-3: 优雅降级机制

| 维度 | 内容 |
|------|------|
| **现状** | LLM 不可用时创建 mockLlm（返回无效数据），非真正跳过 |
| **可行性** | ✅ 可行，需重构 `processTurn` 执行流 |
| **方案** | 每个 LLM 依赖步骤增加 `if (!llmEnabled) { warn + skip; return empty; }` |
| **风险** | 中 — 需修改 extractor 调用链，测试需覆盖 LLM 关闭场景 |
| **工作量** | 1 天 |

### F-4: 结构化日志

| 维度 | 内容 |
|------|------|
| **现状** | 50+ 处 console.log/warn/error，格式不统一，仅靠 BM_LOG_LLM/BM_DEBUG 控制 |
| **可行性** | ✅ 可行 |
| **方案** | 新增 `src/utils/logger.ts`，支持 error/warn/info/debug 四级，渐进式替换 |
| **风险** | 低 — 纯加法，旧日志不受影响 |
| **工作量** | 0.5 天 |

### F-5: 统计指标查询

| 维度 | 内容 |
|------|------|
| **现状** | 无统计 API，5 张表可用 SQL 聚合查询 |
| **可行性** | ✅ 可行 |
| **方案** | 新增 `getStats()` 方法，返回 Stats 对象（节点/边/社区/缓存等统计） |
| **风险** | 极低 — 只读操作 |
| **工作量** | 0.5 天 |

### F-6: CLI 诊断工具

| 维度 | 内容 |
|------|------|
| **现状** | 已有 configure.js/setup-openclaw.js，无诊断命令 |
| **可行性** | ✅ 可行 |
| **方案** | 新增 `scripts/doctor.js`，一键检查环境/依赖/配置/DB 状态 |
| **风险** | 低 — 独立脚本，不影响核心代码 |
| **工作量** | 0.5 天 |

### F-7: 参数化查询安全审计

| 维度 | 内容 |
|------|------|
| **现状** | 82 处 `.prepare().get/run/all()` 查询 |
| **验证方法** | 逐文件检查所有 SQL 查询，确认是否使用 ? 占位符 |
| **结果** | ✅ 100% 参数化 — 所有 CRUD 使用 ? 占位符，db.exec 仅用于 DDL/事务 |
| **调整** | 降为文档化，半天完成审计报告，无需代码修改 |
| **工作量** | 0.5 天（文档化） |
