### 2.3 失败测试 🔴 1 个 (重复性失败)

```
FAIL: test/perf-tiered-benchmark.test.ts > A-3 LPA 阶梯基准 (full vs incremental) > LPA @ 5000 节点 (5% 脏)
AssertionError: expected 159.688ms to be less than 140.725ms (fullTime * 1.05)

结论：增量 LPA 在 5000 节点 + 5% 脏标记场景下反而比全量慢 13%
根因分析：
  - 增量路径的 subgraph walk + PPR 重算开销 > 全量 LPA 开销
  - 5% 脏节点在 5000 节点下 = 250 个种子节点，递归 CTE 遍历 1-hop 邻域代价高
  - 当前阈值 (5% / 5000 节点) 下"全量快于增量"是预期行为，测试应该调整断言条件或阈值公式
```

| 外部 embedding API | ~5 | 测试环境无 embedding 服务 |

**优点**:
- 清晰的 4 层架构，依赖方向严格从上到下
- `IStorageAdapter` 接口抽象让算法层与存储引擎解耦，支持 SQLite/LanceDB 双后端
- 每个功能域独立目录，高内聚
- v1.2.0 引入的 HookRegistry 让外部可以无侵入地扩展提取/召回/融合流程

**隐患**:
- ContextEngine 直接调用 `(this.storage as SQLiteStorageAdapter).getDb()` **打破了抽象屏障** — 是未来切换 LanceDB 的阻断点
- `store/store.ts` (~400 行) 承担了约 90% 的 SQL 操作 + 数据转换，God Object 风险
- 算法层部分函数（如 fusion）仍然直接访问 `this.storage` 的多个方法，而非通过更高层的组合
