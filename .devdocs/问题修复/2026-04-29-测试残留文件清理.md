## 二、根因分析

**2 个测试文件**存在资源泄漏问题：

| 文件 | 根因 |
|------|------|
| `test/context-engine.test.ts` | `afterEach` 仅 `fs.unlinkSync(testDbPath)` 删除 `.db` 主文件，未调 `engine.close()`，未清理 `.wal/.shm` |
| `test/migrate.test.ts` | 使用 `:memory:file:integration_test_<timestamp>` 创建文件 DB，测试完成后未关闭连接、未删除文件 |
| `tests/unit/context-engine.test.ts` | 与第一个问题相同，同一份测试的重复副本 |

**本质原因**：SQLite 开启 `PRAGMA journal_mode=WAL` 后，关闭连接前不会自动清理 `.wal/.shm` 文件。测试代码只删了主文件，忽略了伴生文件。

## 五、经验沉淀

### 测试中 SQLite 的正确清理模式

```typescript
afterEach(() => {
  // 1. 先 close 连接（flush WAL）
  try { engine.close(); } catch { /* ignore */ }
  
  // 2. 删除主文件 + WAL + SHM
  cleanupDbFiles(dbPath);
});

function cleanupDbFiles(dbPath: string): void {
  const files = [dbPath, `${dbPath}-wal`, `${dbPath}-shm`];
  for (const f of files) {
    try { if (fs.existsSync(f)) fs.unlinkSync(f); } catch { /* ignore */ }
  }
}
```

### 教训

1. **`:memory:file:` 不是纯内存** — 会产生真实文件，必须手动清理
2. **WAL 模式下不能只删主文件** — `.wal/.shm` 会残留
3. **`beforeEach` + `afterEach` 双保险** — 防上次测试失败导致残留
4. **重复测试文件** — `test/` 和 `tests/unit/` 下有相同的 context-engine 测试，建议后续合并
