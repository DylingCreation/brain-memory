**林峰**：我做了压测和架构评估。如果纯用 SQLite，部署确实轻量，零运维，但多租户并发写入、复杂联表查询和分布式事务会直接卡脖子。考虑到咱们未来要对接企业客户，数据一致性和扩展性必须优先考虑，我建议**直接上 PostgreSQL**。它的 MVCC 并发模型和 JSONB 灵活查询非常适合这种看板类业务。

**苏晴**：同意。前端做离线缓存时我打算用 IndexedDB，如果后端是 PG，字段映射和同步冲突解决会更规范。不过林峰，初期开发环境能不能先用 SQLite 跑本地 Mock？我想快速搭 UI 组件库联调。

**林峰**：没问题。开发阶段用 SQLite 零配置，测试和生产环境切到 PG。我写一个基于 Prisma 的数据适配层，通过环境变量切换，一行命令就能换库，不阻塞你前端进度。

**周宇**：好，技术选型敲定：**开发环境 SQLite，生产环境 PostgreSQL**。另外，协作流程上咱们怎么配合？苏晴上次提过偏好？

林峰主导后端 API 与 DB 设计，苏晴负责前端状态管理与组件库，我负责需求评审、测试用例和 CI/CD 流水线。大家确认下边界？

**林峰**：分工清晰。不过说个刚遇到的问题：昨天做 M2 预压测时，发现高并发下 PostgreSQL 连接池迅速打满，同步接口出现大量 `504 Gateway Timeout`。

**周宇**：影响面大吗？排查出原因了吗？

**林峰**：已经定位并解决了。主要是两个点：一是几个列表接口存在 N+1 查询，拖垮了 DB CPU；二是默认连接池配置太小。我昨晚引入了 **PgBouncer 做连接池中间件**，复用率提到 90% 以上；同时把同步逻辑里的行级排他锁改成了**乐观锁（基于 version 字段）**，重试机制也加上了。现在 TPS 稳在 600+，P99 延迟压到 80ms。修复代码已合入 dev 分支。
